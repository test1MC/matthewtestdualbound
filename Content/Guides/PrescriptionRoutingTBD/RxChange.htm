Rx Change

The Rx Change Request message is originated by the pharmacy. This message is used to request a change to a new prescription. This request may be used when a pharmacy identifies a need to make a change to the original new prescription.

An Rx Change Response message is used to respond to an Rx Change Request message. An example would be a request for a generic substitution.

The three codes available in the standard for Rx Change are Therapeutic Interchange, Generic Substitution, and Prior Authorization. However, because we have more than three use cases, we have combined several into Therapeutic Interchange as shown below.

Use cases are described below along with how to populate the pertinent fields.

Use Cases – Therapeutic Interchange (Rx Change Request
Type = T)

Use Case #1: DUR (Drug Utilization Review)

When the pharmacy and/or payer detects a DUR event related to the written product, the pharmacy can send an Rx Change Request – Therapeutic Interchange to the prescriber requesting a switch to an alternative drug that will treat the condition with lesser or no adverse effects. Possible interactions include: allergies, dose alerts, drug interactions, and drug diseases.

  • REQ-010 <ChangeRequestType> = T

  • Use DRU-100 <DrugUseEvaluation> of the medication prescribed segment to document further explanation, conflict, or clarification of services related to drug use evaluation.

  • Use DRU-090 <Note> of the medication prescribed segment to indicate DUR event.

Use Case #2: Formulary Compliance Change to an On Formulary or Preferred Drug

When the pharmacy receives a claim reject (or even paid) message which identifies preferred alternative drugs which may have a lower copay for a patient, the pharmacy can send an Rx Change Request – Therapeutic Interchange to the prescriber requesting a switch to one of the preferred products.

  • REQ-010 <ChangeRequestType> = T

  • Use DRU-090 <Note> of the medication prescribed to indicate Formulary Compliance.

Use Case #3: Days Supply Change From 30 Days to 90 Days Supply

When the pharmacy receives a prescription indicating a 30-day supply, but the pharmacy determines that the patient has a 90-day benefit, the pharmacy can send an Rx Change Request – Therapeutic Interchange to the prescriber requesting a switch to a 90-day supply.

  • REQ-010 <ChangeRequestType> = T

  • Use DRU-090 <Note> of the medication prescribed segment to indicate Days Supply.

Use Case #4: Therapy Change – Daily Dose Optimization

When the pharmacy receives a prescription indicating a one 10 MG Tablet twice a day, but the patient desires to take one 20 MG tablet once a day, the pharmacy can send an Rx Change Request – Therapeutic Interchange to the prescriber requesting a switch to one 20 MG Tablet once a day.

  • REQ-010 <ChangeRequestType> = T

Use Case #5: Script Clarification

When the pharmacy is unsure of the prescriber's intent; e.g. Prozac weekly once a day for 200 MG versus 20 MG which looks like a prescriber error. The pharmacy can send an Rx Change – Therapeutic Interchange to the prescriber requesting clarification.

  • REQ-010 <ChangeRequestType> = T

  • Use the Medication Requested segment to indicate what the pharmacy thought the prescriber intended.

Workflow – Therapeutic Interchange

  1. Pharmacist determines which one of the following applies: DUR issue, Formulary Compliance, Days Supply, or Therapy Change, and wishes to request a change.
  2. Rx Change Request is sent for Therapeutic Interchange.
    1. REQ-010 <ChangeRequestType> = T
      1. If the original prescription was sent electronically, the RelatesToMsg will be set to the MessageID of the original prescription.
      2. If the original prescription was sent electronically and included the prescriber order number (PON), it will be sent on the request.
      3. If the original prescription was filled, the pharmacy will send the RxReferenceNumber of the original prescription.
  3. Prescribed medication is sent with the DUR information added in the prescribed drug and a reason in the Notes field. For DUR scenarios, the DUR information is also coded in the DRU-100 <DrugUseEvaluation>.
  4. For each Requested Medication, at least the Drug Description, Quantity, SIG, and Number of Refills is sent.
    • If this is a DUR scenario, then the requested drug segment is not required.
  5. Prescriber receives request – possible responses:
    • Approved (no changes to requested drug)
      1. The prescriber may only send Approved status type if the prescriber selects one of the Medication Requested drug items and all information from the Medication Requested is approved by the prescriber.
      2. If the prescriber selects one of the Medication Requested drug items but wishes to change one of the requested element values (e.g. Directions from Take one Tablet every day to Take two Tablets every day), the prescriber should instead specify Approved With Changes status type (see below).
      3. Approved status type is an indication to the pharmacy to discontinue dispensing the original prescription and instead begin dispensing the new prescription as sent in the Rx Change Response.
      4. When the prescriber responds with Approved, the prescriber should send the Medication Prescribed set to one of the medications requested by the pharmacy.
      5. If no requested drug is sent as in the DUR scenario, the Approved response is not allowed.
    • Approved with changes (anything can be changed on requested medication)
      1. The prescriber should send Approved With Changes status type if the prescriber wishes to approve the change request, but also wishes to:
      2. Prescribe a drug item that differs from one of the Medication Requested drugs
      3. Select one of the Medication Requested drugs AND change one of the requested element values (e.g. Directions from Take one Tablet every day to Take two Tablets every day),
      4. Approved with Changes status type is an indication to the pharmacy to discontinue dispensing the original prescription and instead begin dispensing the new prescription as sent in the Rx Change Response.
      5. Approved with Changes is not to be used when one of the Medication Requested segments is selected and none of the fields are changed. Instead, use Approved status type as described above.
    • Denied
      1. Denied is an indication to the pharmacy to not make a change and instead, to continue dispensing the original prescription.
      2. On a Denied response, no drug information is required to be sent back.
      3. A note or reason code explaining the denial is required to be sent back.
      4. If the prescriber wishes to discontinue the original prescription, then a Denied Rx Change Response should be sent and a Cancel request on the original prescription or follow up phone call should then be made to the pharmacy.
      5. If no Medication Requested is sent as in the DUR scenario, the Denied implies the prescriber is aware of the DUR but wants to continue the original order.
  6. Prescriber sends back response.
    • Appropriate status type is sent
    • The RelatesToMsg will be set to the MessageID of the request.

Use Cases – Generic Substitution (Rx Change Request Type = G)

Use Case #1: Switching From Brand To Generic

When the pharmacy receives a prescription which indicates Substitution not allowed, but a generic is available, the pharmacy can send an Rx Change Request – Generic Substitution to the prescriber requesting that the generic be allowed to be dispensed.

Use Case #2: New Generic Available

When the pharmacy received the original prescription, it was written for a brand drug with no generic alternatives. The prescriber indicates substitution not allowed. During the course of therapy, a generic becomes available; the pharmacy can send an Rx Change Request – Generic Substitution to the prescriber requesting that the generic be allowed to be dispensed.

Use Case #3: Brand Not Covered

When the pharmacy received the original prescription, it was written for a brand drug and the prescriber indicated substitution not allowed. The pharmacy bills the brand, but the payer indicates brand not covered. The pharmacy can send an Rx Change Request – Generic Substitution to the prescriber requesting the generic be allowed to be dispensed.

  • REQ-010 <ChangeRequestType> = G

Workflow – Generic

  1. Pharmacist/Pharmacy application determines there is a generic available and original prescription is marked as substitution not allowed (DAW). If the original prescription is not marked as DAW then the pharmacy can change to a generic without sending this message.
  2. Rx Change Request is sent for Generic Substitution
    • REQ–010 <ChangeRequestType> = G
      1. If the original prescription was sent electronically, the RelatesToMsg will be set to the MessageID of the original prescription.
      2. If the original prescription was sent electronically and included the prescriber order number (PON) that will be sent on the request.
      3. If the original prescription was filled, the pharmacy will send the RxReferenceNumber of the original prescription.
  3. Prescribed Medication from the original order is sent.
  4. The requested medications are sent with the generic(s) that the pharmacist is requested to switch to.
  5. Prescriber receives request – possible responses:
    • Approved (no changes to requested drug)
      1. Approved is indication to the pharmacy to discontinue dispensing the original prescription and instead begin dispensing the new prescription.
      2. The prescriber may only send Approved status type if the prescriber selects one of the Medication Requested drugs and all information from the Medication Requested is approved by the prescriber.
    • Approved with changes

    This value is not allowed for generic substitution. If the prescriber wishes to change the medication, they will need to send a Denied status type as described below, then send a Cancel Rx Request message that the pharmacy will need to approve, and then a NewRx message.

    • Denied
      1. Denied is an indication to the pharmacy to not make a change and instead, to continue dispensing the original prescription.
      2. On a Denied response, no drug information is required to be sent back.
      3. A note or reason code explaining the denial is required to be sent back.
      4. If the prescriber wishes to discontinue the original prescription, a Denied Rx Change Response should be sent and a Cancel Rx Request on the original prescription or follow up phone call should then be made to the pharmacy.
  6. Prescriber sends back response.

    • Appropriate status type is sent.

    • The RelatesToMsg will be set to the MessageID of the request.

Use Case – Prior Authorization (Change Request Type = P)

When the pharmacy receives a claim reject indicating that a Prior Authorization is required, the pharmacy can request a Prior Authorization from the prescriber.

  • REQ-010 <ChangeRequestType> = P

Workflow – Prior Authorization

  1. Pharmacist determines that Prior Authorization is required for this prescription. This is usually done by submitting the claim and getting a reject for prior auth.
  2. Rx Change Request is sent for Prior Authorization.
    • REQ-010 <ChangeRequestType> = P
      1. If the original prescription was sent electronically, the RelatesToMsg will be set to the MessageID of the original prescription.
      2. If the original prescription was sent electronically and included the prescriber order number (PON), it will be sent on the Rx Change Request.
      3. If the original prescription was filled, the pharmacy will send the RxReferenceNumber of the original prescription.
  3. The COO <BenefitsCoordination> information is sent so the prescriber knows who to contact.
  4. Prescriber receives request – possible responses:
    • Approved
      1. The prescriber has obtained prior authorization from the Payer for this medication.
      2. The medication prescribed segment is sent back to the pharmacy in the Rx Change Response.
      3. If a prior authorization number is required by the payer, this number should be included in the medication prescribed segment.
    • Denied (Payer Contacted)
      1. The prescriber contacted the payer but the payer denied the request.
    • Denied (Payer not Contacted)
      1. RES-020 <DenialReasonCode> No attempt will be made by the prescriber to obtain Prior Authorization. This is an existing code (AO) in the NCPDP external code list.
    • Denied
      1. On a Denied response, no drug information is required to be sent back.
      2. A note or reason code explaining the denial is required to be sent back.
      3. The original prescription remains valid.
  5. Prescriber sends back response.
    • Appropriate status type is sent.
    • The RelatesToMsg will be set to the MessageID of the request.

Rx Change General Message Flow

The following steps depict the Rx Change message flow:

  1. A NewRx is processed by the pharmacy and the pharmacy identifies that a change to the prescription may be appropriate.
  2. Pharmacy submits the Rx Change Request via Surescripts to the original prescriber.
  3. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  4. Surescripts successfully transmits the Rx Change Request to the prescriber via the prescriber system.
  5. Prescriber system validates the Rx Change Request and returns a Status or Error (if errors are found) message.
    1. If a Status 000 is sent in step 5, the prescriber system would send a Verify.
  6. Surescripts transmits the Status, Error, or Verify message back to the pharmacy system.
  7. Prescriber system processes the Rx Change Request and returns the prescriber's Change Response message.
  8. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the prescriber system.
  9. Surescripts transmits the Rx Change Response message back to the pharmacy system.
  10. Pharmacy system validates the Rx Change Response and returns a Status or Error (if errors are found) message.
    1. If a Status 000 is sent in step 11, the pharmacy system would send a Verify.
  11. Surescripts transmits the Status, Error, or Verify back to the prescriber system.

Rx Change Best Practices

  1. Notes specific to the reason for the request should go in the prescribed drug segment as defined in the use cases above. Notes specific to all drugs requested should go on the first requested drug segment.
  2. The COO Benefit segment should be included so the prescriber knows which payer the pharmacist is using to determine coverage/pricing.
  3. If the Rx Change for Therapeutic Interchange or Generic Substitution is approved, this replaces the original Rx. The prescriber should create a new PON (prescriber order number); the old one becomes inactive for this approval. If the Rx Change for Prior Authorization is approved, the original prescription can still be used.
  4. A (RxChange) denial does not make the prescription invalid or stop therapy. If the prescriber wishes to stop therapy, the prescriber should send a cancel message for the original NewRx.
  5. An Approved with Changes Response (RES-010 = C) should not be used for Prior Authorization request; it has been removed in a future version of SCRIPT.
  6. Even though the request segment is optional, it should be sent to identify the type of request and it also should be returned on the response.
  7. A pharmacy may send an Rx Change Request for controlled substances (EPCS) regardless of whether the prescriber is setup for the EPCS service. For approvals, when received, if the prescriber is not enabled for the EPCS service, the prescriber should deny the electronic request with status type "denied" and indicate in the free text field that the denial will be followed by a written or telephoned prescription order. If the prescriber is set up for the EPCS service they can approve the change request as stated in the requirements. Denials should occur as stated above in Section 5.3 for the specific use case that applies.
  8. The free text field (RES-040) should be accessible for entry and accessible for prescriber viewing (added per advisory group notes).
  9. If the prescriber approves the change response it should be treated just like a new prescription. With a change response there are no limits or requirements around what can or cannot change in the message like there are on a renewal response. The pharmacy system should absorb the response data like they would on a new prescription in order to capture all the necessary information the prescriber sent.
