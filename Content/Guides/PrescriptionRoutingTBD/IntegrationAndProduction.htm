Integration Process

Implementation of EPCS

Only qualified application vendors and pharmacies that have: completed Surescripts Certification, have a third-party Proof of Audit Letter, and have a Surescripts EPCS Audit Attestation Form, will be allowed to connect to the Surescripts network for EPCS.

The following steps are required to implement EPCS:

1). Update e-prescribing software to meet all requirements specified by the DEA in the Interim Final Rule and the SCRIPT messaging version (this guide) that supports EPCS.

2). Undergo a third-party audit to ensure the software meets all DEA EPCS requirements.

3). Achieve Surescripts Certification to the Surescripts Implementation Guide.

4). Make audit results available to Surescripts with a Proof of Audit Letter & Surescripts EPCS Audit Attestation Form.

Prescription Routing General Best Practices

The following general best practices apply to all prescription routing message types.

1. The application should make clear to the user, which data elements will be sent to the receiver and whic elements are for interna/local use.

2. If customers implement any type of prescription message store and forward methodology, e.g. hub or central location, the storage/hold time period should be no more than 1-2 minutes.

3. to maximize the use of electronic transmission, customers should default to electronically send within their application workflow when the receiver is electronically enabled.

4. Application should not duplicate messages; i.e. electronically delivered as well as faxed.

5. Applications should have the ability to send ICD-9 or ICD-10 information in all message types. This is because ICD-9/ICD-10 coding is required on some electronic claims. If the information is not included, it could result in a manual process between the provider and pharmacy.

6. Application should have the ability to send or receive EffectiveDate for future dating prescriptions.

7. Notes and other free-text fields should not contain information for which there is a designated field.

Certification Requirements

The Surescripts certification process ensures that customer software can send and receive electronic messages in accordance with industry standards and that it provides open choice for medication selection and dispensing location. In addition, Surescripts certification focuses on patient safety, efficiency of the electronic prescribing process and ease of use by end-users.

Surescripts certification testing focuses on message format, application workflow, and display in accordance with Surescripts Implementation Guides and the associated Application Certification Requirements (ACR) document noted in Document References on page 16. By holding all customers accountable for meeting the ACRs, our customers can send and receive the highest quality messages as e-prescribing and clinical messaging continue to progress overall as an industry.

Requirements that are enforced as part of the production code are denoted as “must” and will need to be met in order to successfully complete certification. “Should” is used for guidance or best practices. See the following chart for terminology usage in this implementation guide.

Surescripts understands our customers are in the best position to make workflow design decisions and we are committed to encouraging innovative designs. Surescripts is also committed to ensuring the highest level of patient safety possible. If a workflow could potentially result in a patient safety issue, or if a patient safety issue is identified, Surescripts may require application code changes to ensure positive patient outcomes.

To ensure patient safety, the application shall take steps to ensure that the following critical fields (as will be transmitted in the message) are reviewed by the sender for accuracy and displayed to the receiver.

The code value for codified fields does not have to be displayed, only the description.

  • Patient First Name
  • Patient Last Name
  • Gender
  • DOB
  • Drug Description
  • Quantity/PUC
  • SIG (free text sig vs. Structured)
  • Notes (to/from pharmacy)
  • Refills
  • Days Supply
  • Pharmacy name
  • Effective Date (if in future)
  • Height and Weight, if applicable

Any element that is not sent with the transaction shall*106R-041 be labeled “For internal use only” on the summary screen, so the prescriber knows that information will not be sent with the message.

When a field is not sent, it shall*106R-155 not be defaulted to display 0.

To allow for patient choice, the application shall allow a pharmacy search based on pharmacy type of mail/retail at a minimum; other pharmacy search options may include specialty and long term care.

The application shall not filter out pharmacies of a specific type when a search by geography (city, state, and zip code) is performed, unless a specific pharmacy type is requested. The application shall display the following pharmacy information in the pharmacy search results: Pharmacy Name, Street address, City, State, Zip code, phone number and pharmacy type.

Communication Rules

Timeouts

For timeouts, consider the following:

  • • When sending a message to Surescripts, the initiator should set the http timeout to no less than 30 seconds.

  • • A receiving system must reply with a Status/Error response within 24 seconds.

  • • When a Status 000 is sent as a reply, a receiving system must follow-up with a Verify or Error message within 24 hours because Surescripts may return a timeout error.

  • • To avoid duplicate requests, sending systems should wait at least 48 hours before retrying RefillRequest if no RefillResponse has been received.
