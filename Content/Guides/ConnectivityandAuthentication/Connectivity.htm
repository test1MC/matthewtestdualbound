Connectivity

Connectivity Methods

HTTPS

Surescripts supports the HTTPS method for connectivity to transmit transactions between customers. Using HTTPS, customers act as the client when initiating and submitting POST web requests to the servers on the Surescripts system. With certain transactions, Surescripts act as the client when initiating and submitting POST requests to servers on customer systems

Surescripts supports HTTPS with the following specifications:

  • TCP/IP is the communication protocol utilized between the customer and Surescripts.
  • Customer is required to use a static, registered IP address.
  • When Surescripts initiates an HTTPS session with a customer, the customer’s server must use a digital certificate with, at minimum, a 2048-bit public key. This public key is used to encrypt, at a minimum, a 128-bit session key; however, 256-bit is preferred.
  • For server certificates, Surescripts requires a public Certificate Authority (CA) and does not accept self-signed certificates.
  • Surescripts requires Transport Layer Security (TLS) 1.2 with the HTTPS protocol. Surescripts restricts the use of certain cryptographic algorithms and protocols. Weaker cipher suites export grade are not supported.

Requirements to Connect to Surescripts

Customers connect using HTTPS over the Internet.

  • Surescripts filters access to web services by IP address. Customer must provide a static IP address (or a list of IP addresses) for all TCP/IP traffic to Surescripts.
  • Customer request to production and staging cannot originate from the same source IP address. The customer must provide two static lists of IP addresses; one for staging and one for production. These static lists of IP addresses provided by the customer for staging must be different than the address provided for production.

This is critical since the staging environment must never contain actual Protected Health Information (PHI), and the production environment should not contain test information. Customer employees/contractors may have varying levels of access to PHI; therefore, access to Staging/QA/Development systems should not directly allow access to production.

  • Surescripts permits up to six (6) separate customer-owned IP addresses per Surescripts environment.
  • Customers must use the Surescripts-specified FQDN when configuring their system. Do not use IP addresses in host files or within application software. Failure to comply will result in an inability to transmit messages to Surescripts data centers. Surescripts employs load balancing which relies on the specified Surescripts FQDNs to route traffic.
  • The IP address provided must either be registered with the American Registry for Internet Numbers (ARIN) using the customer's name and contact information (i.e. Amazon, AT&T, Yahoo, or Comcast should not be used), or the customer must send an email to Surescripts to affirm that the customer-provided IP address is operated by the customer. The email must come from the customer’s domain (i.e. "@yahoo.com" or "@gmail.com" may not be used).
  • TLS connections inbound to Surescripts must use TCP/IP port 443.
