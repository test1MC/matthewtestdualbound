Prior Authorization Integration & Production

The following section defines related technical and support elements needed to achieve certification for moving into production, along with our Integration, Compliance, and Certification processes.

Integration Process

Once Surescripts assigns an Integration resource, customers will be contacted to begin the project(s) and will be guided through the Surescripts Integration process. The Integration resource will provide Surescripts documentation, product education, access to the Surescripts staging environment, assistance during design and development phases and testing to prepare for certification. The Integration resource will perform certification testing, and, once successful, will provide access to the Surescripts production environment.

The time frame of the Integration process can vary depending on your resource allocation for the project.

Meeting Requirements

During Integration, customers undergo certification testing to demonstrate all requirements are met. Certification focuses on message format, and, when appropriate, application workflow and display in accordance with Surescripts documentation and the associated Application Certification Requirements (ACRs). Upon successful completion of certification and, when applicable, other pre-production network requirements (e.g. Identity Proofing, 1311 audit), customers may transition to production.

For requirements, consider the following:

  • Surescripts ACRs are required to be met to achieve production status on the Surescripts network and will be enforced as part of certification.
  • Surescripts business rules are data validations applied by Surescripts beyond the schema requirements that will cause a message to be successful or rejected.
  • Surescripts test cases do not cover all possible scenarios in production. Customers are responsible for testing all scenarios specific to their production environment.
  • Customers must be prepared to make allowances for things that are not governed by our ACRs or NCPDP requirements or recommendations.

Connectivity HTTPS

The Surescripts network uses the HTTPS protocol for the transport of messages between network customers. HTTPS is a secure, reliable, and widely used protocol for the exchange of information over TCP/IP.

Communication Rules

In order for the network to be reliable, there are communication rules to which all customers are required to adhere.

Security Rules - Surescripts requires that all messages sent and received use Basic Authentication. The "Authorization" header of the HTTP message must contain the user-ID and password, separated by a single colon (":") character, within a base64 encoded string. For example: Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==

How Basic Authentication Is Used

Surescripts sets up a username and password for customers to use in messages that customers send to Surescripts. Customers will always use Basic Authentication in the message header for messages posted to the Surescripts assigned URL in test and production. The assigned URL ends with "/AuthenticatingXMLServer.aspx". For example: https://messaging.surescripts.net/CustomerName/AuthenticatingXMLServer.aspx

Attaching the username and password to the header (Base64 password encoding)

Concatenate the username and password, separating them with a colon: "username:password"

Convert that concatenation into a UTF8 encoded byte array

Convert that array into a BASE64 encoded string:
dXNlcm5hbWU6cGFzc3dvcmQ=

Specify the authentication type of 'Basic' by adding the word 'Basic' to it:
"Basic dXNlcm5hbWU6cGFzc3dvcmQ="

Add that string to the Authorization header. It will look like:
Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=

Send the web request - HttpWebRequest.Post

Timeouts

For round-trip messages, the initiator of a message can expect Surescripts to timeout after 27 seconds of attempting to respond to the request. Therefore, the initiator should set their time-out to a value at least three seconds greater, or 30 seconds.

Provider Vendor Best Practices Workflow

  1. When sending a message, set the timeout to 30 seconds.
  2. Surescripts receives the message and takes up to 3 seconds to process it and send to the receiving customer.
  3. Surescripts’ timeout window is set to 27 seconds (30-3)
  4. The receiving customer should send a Status/Error response back within 24 seconds.
  5. Surescripts receives the response message from the receiving customer and takes up to 3 seconds to process it (27-3).

Equation

30 - 3 - 3 = 24

Three (3) seconds for Surescripts to process the incoming message from sender; three (3) seconds for Surescripts to process the outgoing message back to the sender = 24 seconds timeout for receiver.
