Message Linkage

In order to link messages together, four key fields are defined below. This linkage allows a software system to tie one message to another. For example, on a PAInitiationResponse the PBM/payer system needs to find the PAInitiationRequest to which the provider vendor is referring. This linkage should be done in an automated manner.

MessageID – The unique number assigned to each message.

RelatesToMessageID – Is used in the conversation (message string) to link the current message to the most recent message received from a trading partner. In RelatesToMessageID use the latest MessageID in the conversation (message string) received from a trading partner that was not a utility message (for example, not a GetMessage, Status, Verify or Error).

PAReferenceID – Assigned by the prescribing system on the initial message and is used by the prescribing system as a tracking identifier on all request and response prior authorization messages. The identifier must be unique per prescribing system.

PACaseID – PBM/Payer-assigned identifier for this request and used by the PBM/payer as a tracking identifier on all request and response prior authorization messages.

Third Party Processor Message Linking

When PBM/payers return a CO or CP reason code, the message becomes internally marked as closed for 3rd party processing. If Surescripts’ 3rd party processor search finds a match, a secondary PAInitiationRequest is created based upon the initial PAInitiationRequest. The recipient is updated and the MessageId from the original PAInitiationRequest is cloned and modified with a prefix of EPAINI. When the 3rd party processor responds, the message will have a RelatesToMessageId that has a prefix of EPAINI and references the secondary PAInitiationRequest.
