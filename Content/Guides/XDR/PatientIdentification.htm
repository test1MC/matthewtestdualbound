Patient Identification

A patient identifier is a requirement of the IHE IT Infrastructure Technical Framework, Volume 3, Revision 12.2 Final Text 22 April 2016 for XDR Requirements.

Surescripts inserts patient identifiers/demographics found in the message into the XDR message sent to the customer XDR interface using the following priority:

  1. Patient demographics in the attached clinical document (CDA or CCR)
  2. XDR metadata found in the XDR envelope or the XDM metadata file
  3. Patient demographics in the header

If there are multiple clinical documents, the first document is used as the source of the patient identifier.

If there is no patient data available in the incoming message, Surescripts passes along the message without populating the patient data section of the XDR.

By using the exact patient identifier or a demographics-based match, incoming messages can be associated with the correct patient or routed to the appropriate triage queue.

Surescripts recommends the customer obtain an Object Identifier Definition (OID) from a registration authority. OIDs are a preferred scheme for unique identifiers in HL7 and should always be used except if one of the inclusion criteria for other schemes apply.

Because patient IDs are passed as-is from multiple parties, Surescripts cannot guarantee that patient IDs are unique. Customers must ensure their workflows allow for verification of all patient matches based on third-party IDs, and allow users to correct patient mismatch cases when the same ID might have been used for multiple distinct patients.

XDR interfaces may have different levels of support for XDR message content. A typical Surescripts network message is an email-like message that contains a message body (free text) and a clinical document attachment (such as a CDA).

Furthermore, customers integrating with the XDR interface are encouraged to accept as wide a variety of content types (e.g. PDFs, CDAs) as possible. Each customer XDR interface can be configured with the acceptable content types. Surescripts informs senders of messages with unsupported content that their message was delivered successfully. This separate message is generated by Surescripts detailing which attachments were omitted from the delivered message.

Constraints on Standard XDR

The following constraints are placed on the ITI TF-2b section 3.41 definition of XDR:

  • The transaction shall carry at least one document.
  • XDS Folder definitions are not supported.
  • Only new document submissions, without reference to previous submissions, are supported (e.g. there is no support for replacing documents sent in previous transactions, or referencing documents from previous transactions).
  • The Basic Patient Privacy Enforcement Option ITI TF-20 3.41.4.1.3.1 is not supported.
  • The request/response must use the synchronous web services exchange model.
  • Customers must follow all requirements in XDR for Direct.
