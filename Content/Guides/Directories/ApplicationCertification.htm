Application Certification Requirements

As the source data can be slightly different in respective internal and external local databases, Surescripts provides the following definitions as support to the ACR:

  • STANDARDIZED RECORD
    Shall be interpreted as an address record in which the intent of the physical location is not altered, but the formatting/styling/abbreviations of field such as directional (North vs N, South vs S., etc.) and location designations (Suite, Flr, etc.) are abbreviated and/or may have punctuation removed.
  • ENHANCED RECORD
    Shall be interpreted as content from an alternate credentialing source (e.g. NPPES or DEA) for a specific provider at a specific practice location.
  • COMPOSITE RECORD

    • A Composite record takes enhanced and standardized record elements and builds a “best possible” representation of the provider at that location. Some elements, such as phone and fax number, may be different.

  • THIRD-PARTY DATA SOURCE
    An alternate, third-party data source is defined as a credentialed entity or agency (e.g. NPPES, DEA, State Board) or a Surescripts recognized commercial entity that uses credentialed sources to develop an enhanced/composite record. Composite records may include enhanced sourced data, but should not change nor conflict with the intended physical address representative for the Provider.
  • END POINT – defined as the originating user who initiates (sends) a message or the “final” recipient of a message.

D.001 Receivers shall not reject any incoming messages due to misalignment of local directory contents, lack of entries for the sender, or use of address standardization within the customer’s local directories.

D.002 Customers shall ensure that modifications in their customer system to corresponding Provider or Organization directory information are sent to Surescripts within 24 hours.

D.002 Notes/Examples: Modifications include, but are not limited to:

  • Service Level (the ability to accept/receive transactions)
  • Address Demographics
  • Legal Name

Keeping the local directories in sync with the Surescripts Directory drives reduced errors and minimizes delay in patient care.

D.003 Customers shall integrate a process to download the Nightly or Full version of the Surescripts download file daily. Customers must use the DirectoryDownload message type for these processes.

D.004 If downloading the Nightly version daily, the customers shall integrate a weekly download of the Full version.

D.005 Customers shall have a process to ensure that all new and updated Surescripts Directory information, as received in the download, is available and used for message routing within 24 hours.

D.003 - D.005 Notes/Examples: Customers are required to certify to download files using the Directory Messaging API service; manual retrieval of a Download file is no longer supported. The manual approach (using Admin Console) to Directory Record Management will continue to be supported, provided that the requirements of updates to occur within 24 hours are met.

Recommendations: Pulling a Full file daily delivers all of the records and is better suited not to miss anything if an error with a daily Nightly file is encountered.

If the Nightly file is pulled daily, then a process to capture the Full file and apply it against the dataset should be put in place; this will assist in “truing-up” the data, especially if a daily Nightly file error occurs. The information should reach the end user and be made available to them.

Records that are no longer enabled for activity on the network should be removed in a timely manner.

D.006 Every provider or organization record within the downloaded Surescripts Directory shall be made available and considered for messaging. This includes when multiple records may have the same physical address, phone and/or fax.

D.007 Prior to sending a message through Surescripts, the sending customer shall ensure that the intended receiver is enabled with the appropriate service level to accept the electronic transaction.

D.008 The sender shall not send a message when the recipient’s ActiveStartTime is in the future or the ActiveEndTime is in the past.

D.008 Notes/Examples:

At times a provider or pharmacy may have listings at the same physical address, but they represent a different business entity or product and may have a different routing identifier.

Examples:

  • Retail vs. LongTermCare - based pharmacy records
  • Individual/multiple provider records set up with a distinct SPI (for example one SPI with New/EPCS; a second SPI with New/Refills; a third SPI with New/RxCancelCancelRx)

The customer must be able to support records with the same/similar demographics and not relegate similar records to fax or removal from the end user’s directory.

D.009 When Organization details are included within a message transaction on the Surescripts network, the following fields shall match the information in the Surescripts Organization Directory Download for the sender and/or recipient segment:

  • NCPDPID
  • Store Number
  • NPI
  • PayerID
  • Business (Organization) Name

 

D.009 Notes/Examples: A friendly business/organization name for the pharmacy organization may be established but may not replace or eliminate the pharmacy organization name as supplied in the Surescripts Directory and shall not be sent in the outgoing message.

  • Address (Standardization permissible)
  • Phone (Primary preferred, Secondary accepted)
  • Fax (Primary preferred, Secondary accepted)

D.010 When Provider details are included within a message on the Surescripts network, the following fields shall match the information in the Surescripts Provider Directory Download for the sender and/or recipient segment:

  • SPI
  • NPI
  • Provider First Name
  • Provider Last Name
  • Address (Standardization permissible)
  • Phone (Primary preferred, Secondary accepted)
  • Fax (Primary preferred, Secondary accepted)

 

D.010 Notes/Examples: Field content consistency when using multiple datasets is critical.

It is critical that key and routing identifiers not be altered in regards to credential or address demographics; this could result in an unintended sharing of PHI and/or delay in patient care.

Composite and enhanced records may be used provided they do not contradict nor conflict with the Surescripts Directory listing.

In regards to communication numbers, the primary number in the Surescripts Directory is always preferred, but secondary numbers provided against a record within the directory are also acceptable.

If the Surescripts Directory listing does not have the required field for a specific message (for example Pharmacy NPI for a NewRx message), the sender may use local directory or third-party information to ensure the message is complete.

S.103: The sending customer shall send an address within the message which matches the associated Directory record. Note: When a provider system utilizes the Surescripts Learning Directory service, this requirement is satisfied.

The address of the provider of the prescription order should be the practice physical address in which the patient encounter took place. For LTC based encounters, the provider should use the facility based location, or, for home-based service, their practice of record for prescription orders.  For Telehealth based encounters that are not part of an associated provider practice location, the practice address within the message, and correlating directory record, should be associated with a provider associated credentialed location in the State from which the provider is licensed.

On an RxRenewalResponse the provider should change this address to match what is in the Surescripts directory. For address element formatting and consistency information, customers should use the United States Postal Service (USPS) 28 standard as a guideline.
