Using Web Service API

The Web Service API allows access to functionality via RESTful web services for customers who prefer to develop their own user interface.

General Conventions

The Web Service API follows Representational State Transfer (REST) architecture principles using HTTP requests with appropriate HTTP verbs and status codes.

The message body for requests and responses are of type application/json or application/xml.

Versioning and URLs

URLs are of the form https://{{host}}/uiservice/api/{{version}}/{{path}} or , https://{{host}}/mmpoc/api/{{version}}/{{path}} where:

  • the host depends on the environment and is provided  as part of the Surescripts integration process;
  • version is the requested version of the endpoint and is provided as part of the Surescripts integration process
  • path is the path to the specific service

For example: https://{{host}}/uiservice/api/v0/worklist

The API version number ("v0" in the above example) is incremented when a breaking change is introduced. A breaking change prevents an existing implementation from communicating with a particular version. Adding additional fields is not considered a breaking change. An API client application needs to support additional fields being added to an existing version.

Examples of breaking changes includes, but are not limited to:

  • Changing type or attributes of an existing field
  • Changing an endpoint path
  • Changing the semantic meaning of an existing field

Worklist Summary

The worklist/summary provides summary information of task counts for a provider or a patient. The API takes an array of recipients or a patient and a products object, which can include all or just one product. If product is not sent, the customer will get back the task counts for all products implemented.

Workflow Considerations

The worklist summary API is lightweight and useful when only the count of messages is displayed to the user such as on a dashboard.

The diagram below is an example of where the worklist/summary API could be used in a typical patient visit workflow.

Adherence Profile Summary

The adherence-profiles/summary provides the member’s lowest PDC score and associated disease class. The patient’s PBM Unique ID and payer Id obtained from an eligibility transaction are passed as parameters on this request.

Workflow Considerations

The adherence-profiles/summary API is lightweight and useful when only the lowest PDC score is displayed to the user such as on a dashboard.

The diagram below is an example of where the adherence-profiles/summary API could be used in a typical patient visit workflow.

 

The response to this API will return the patient’s lowest PDC value and disease class.

PDC values are categorized as:

  • PDC  value  <= 60%, low
  • PDC value  > 60% and < 80%, moderate
  • PDC value => 80%, healthy

Communication Request

The communication request API retrieves a single message and the associated questionnaire.  The response includes the endpoint link to be used when posting the QuestionnaireResponse. The QuestionnaireResponse endpoint link should be cached in the provider vendor’s system and used as the endpoint when the end user submits a response.

Workflow Considerations

It is recommended to show the medication management message detail, questionnaire and the adherence profile in areas of the workflow where the provider will take action on non-adherence or gaps in care. 

  • The CommunicationRequest API response contains the medication management message detail. 
  • The Observation API response described in the next section of this document contains the member’s adherence profile (PDC scores).

The diagram below illustrates where the CommunicationRequest API should be used in a typical patient visit workflow.

QuestionnaireResponse

The QustionnaireResponse API is used to submit a completed questionnaire back to the PBM/payer for a medication management message (High Risk, Medication Adherence or Missing Medication.  The QuestionnaireResponse endpoint link should be cached in the provider vendor’s system from the CommunicationRequest. The task Id passed is also obtained from the CommunicationRequest API.

Workflow Considerations

It is recommended to show the questionnaire with the medication management message detail and the adherence profile in areas of the workflow where the provider will take action on non-adherence or gaps in care. 

  • The CommunicationRequest API is used to get the message details and questionnaire. 
  • The Observation API is used to get the member’s adherence profile (PDC scores).

The diagram below illustrates where the QuestionnaireResponse API should be used in a typical patient visit workflow.
