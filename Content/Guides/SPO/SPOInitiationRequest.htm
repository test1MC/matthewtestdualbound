SPO Initiation Request

Message Header

The message header follows the NCPDP SCRIPT standard version 2013101.

Message Header XML Diagram

In XML diagrams, blocks that are solid are mandatory, whereas blocks that have dotted lines are conditional.

Message Header Elements

Requirement Designation

Elements that are grouped together may be marked as mandatory; however, if the group itself is marked as conditional, then these are only required if you use the group.

Element Attributes:

Code

Description

M

Required/Mandatory - the element must be used per the specification (e.g., XML schema validation).

R

Required - the element must be used per the business rule.

C

Situational/Conditional - the element must be used if conditions are met. Some fields may not have specific conditions. Data should be sent if available.

Where comments are "Not used by Surescripts"/"Not used by SPO", information will be passed on, but not used for processing by Surescripts.

D

Dependent - the element is required or conditional based on the message type.

This Guide only includes data elements where Surescripts has specific requirements or further explains the field usage. Refer to the NCPDP SCRIPT Schema listed in Document References for a complete list of fields.

M/C

Element

Comment

M

 SPOReferenceID

The unique ID generated by the sender for the SPOInitiationRequest.

C

BenefitsCoordination/PatientInsuranceCard

A scanned image of the patient’s insurance card (front and back of card) can be included in the BenefitsCoordination segment as a .png attachment embedded in within the message.

C

BenefitsCoordination/MedicalBenefitsCoordination

In addition to pharmacy benefits information, medical benefits information can be provided.

M

Patient

 

R

Identification/MedicalRecordIdentification
   NumberEHR

The identifier that uniquely identifies the patient. If this identifier is not unique across all provider vendor installations, use the Identification/MutuallyDefined field in combination with this ID to make a globally unique patient identifier.

This field is optional in the schema, however it is required by SPO. This field will be used in the patient ID field in the Auth Token Request.

Note: If this field is not sent an error will be returned.

R

Identification/MutuallyDefined

Qualifier the provider vendor may use to make their patient ID unique across different organization/facility installations. This is used in conjunction with the Identification/MedicalRecordIdentificationNumberEHR field. This field will be used in the organization field in the Auth Token Request.

Note: If this optional field is not included in the SPOInitiationRequest, it should not be included in the Auth Token Request for the specified patient.

M

MedicationPrescribed

This is modeled after the ePA Medication Prescribed segment; multiple segments are supported. Each segment must include the associated PrescriberOrderNumber.

M

PrescriberOrderNumber

The prescription number assigned by the prescribing system. This will be the same number as what is sent in the NewRx and links the NewRx to the SPOInitiationRequest.
