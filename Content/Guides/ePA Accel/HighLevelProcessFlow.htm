High-Level Process Flow

The following diagram displays the high-level process flow for the Accelerator.

Figure #1: Electronic Prior Authorization CompletEPA® Accelerator Process Flow

  1. The provider vendor sends a PAInitiationRequest to EPAINI, which is the Surescripts identifier for ePA.
  2. Surescripts forwards the message on to the receiving customer. Surescripts will always send the first PAInitiationRequest to the PBM/payer identified in the PayerID if the PBM/payer is enabled for ePA. If the PBM/payer is not enabled for ePA, Surescripts will search for a third party processor. The PBM/payer or third party processor will determine if the patient and drug combination requires PA.
  3. The PBM/payer or a third party processor returns the PAInitiationResponse indicating:
    • Open status (a question set), or
    • Closed status

      • See the Electronic Prior AuthorizationCompletEPA® Companion Implementation Guide for Closed Reason Code information in the Message Business Flow Response Summary, or

    • Closed status when the PBM/payer or a third party processor is not the PA processor.

      In cases where the payer indicates they are not the PA processor, Surescripts will initiate a search to find a different ePA processor. If no route is found to an ePA processor, Surescripts will either:

      • Create a worklist task for the provider from the most recently received PAInitiationResponse indicating that no processor could be found and a manual process should be followed, or
      • Create a workflow task with a URL in the PANote field that the user can select to launch a fillable PA form via PDR.

          Once the user fills out the PA form provided by PDR, PDR faxes the form to the PBM/payer and there will be no further electronic communication to the EMR by either PDR or Surescripts. The PBM/payer will fax the provider vendor back the approval or denial. See the Electronic Prior Authorization Companion CompletEPA® Implementation Guide for more information on PDR.

          The provider vendor will receive one “acknowledge closed PA” task based on the most recent response.

  4. The provider vendor application shows the Accelerator worklist.
  5. The provider selects a task in the worklist and completes the question set.
  6. The Accelerator builds a PARequest from the completed question set and sends it to the PBM/payer on behalf of the provider.
  7. The PBM/payer processes the PARequest and sends a PAResponse to the provider.
  8. Surescripts intercepts the PAResponse and creates a task for the provider to acknowledge the response.
  9. The provider reviews the result of the PAResponse in the Accelerator.
