MPI File Load Overview

The MPI File Load is used to load the participant’s patients that have given consent into the Surescripts MPI.

If a patient opts out of information sharing, exclude or inactivate (using an End Date) the patient from the MPI File Load.

MPI File Load Process Flow

Figure #1 MPI File Load Process Flow Diagram

Initial MPI File Load Process

The following steps depict the flow of the Initial MPI File Load process:

  1. The participant creates a file of patient demographic information for patients that have provided consent to be part of the Surescripts MPI.
  2. The participant submits the initial patient file load to Surescripts via SFTP.
  3. Surescripts populates the MPI with the patient demographic information.
  4. Surescripts creates a response file indicating success and failure details.
  5. The participant pulls the response file via SFTP.

Daily MPI File Load Update Process

The following steps depict the flow of the Daily MPI File Load Update process:

  1. The participant creates a Daily Update file utilizing the same format to keep the Surescripts MPI current.

The Daily Update file should be sent when:

    a). There is a new patient or a change in a patient’s demographic data that Surescripts has defined in the file layout. (If other patient information not contained in the file layout changes, then no update should be sent.)

    b). Any patient opts out of the program. In this case, the participant should include the date on which the patient’s participation will end in the End Date.

  1. The participant submits the Daily Update file to Surescripts via SFTP.
  2. Surescripts processes the updates.
  3. Surescripts creates a response file indicating success and failure details.
  4. The participant pulls the response file via SFTP.

Character Set

Alphanumeric data must only contain ASCII characters with decimal values of 32 through 126, excluding the pipe (hex7c) delimiter character and line feed (hex 0a). All characters outside this range will be rejected. See http://www.asciitable.com/ for a list of valid characters.

Numeric data is composed of the characters 0-9.

Data should be utf-8 encoded.

Situational/Conditional - the segment must be used if conditions are met.

File Naming and Formats

The recommended file naming convention is:

  • Application:
    • RLE for National Record Locator Service
  • Participant Name
  • Date in YYMMDD format
  • XX where XX is a unique sequence within day, if required

RLEParticipantNameYYYYMMDDXX

It is strongly suggested to compress the file prior to transmission to Surescripts. The current zip protocols that are supported are gzip, bzip2 and zip.

  • If compression is used, the archive file should contain only a single input file.
  • The compressed archive file should have the same file name as the file it contains, with the addition of the file extension of the compression method.

Example: File name: RLEAcmeHealthCare2014022001.txt is then placed in a compressed archived: RLEAcmeHealthCare2014022001.txt.gz

The input file must be delivered to Surescripts using FTP and placed in the “to_surescripts” folder.

The response file from Surescripts will be placed in a folder named “from_surescripts”.

The response file name will be based on the original input file name.

Example:

Original input:     RLEAcmeHealthCare2014022001.txt

Response:     RLEAcmeHealthCare2014022001.rsp

Compressed response: RLEAcmeHealthCare2014022001.rsp.gz
