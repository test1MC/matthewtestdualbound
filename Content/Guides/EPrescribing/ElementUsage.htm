RxRenewal Requirements

Exceptions to Fields that Must be Returned

If the field is not listed as optional to update, it must be returned in the response; however, Structured Sig, CompoundInformation, and Facility are three exceptions:

  • The structured portion of Sig element is optionally supported. If received on an RxRenewalRequest, it is not required to be sent back on an RxRenewalResponse. The application should instead use the SigText from the Sig element.
  • The ComnpoundInformation element is optionally supported. If received on an RxRenewalRequest, it is not required to be sent back on an RxRenewalResponse.
  • The Facility segment is optionally supported. If sent on an RxRenewalRequest, it is not required to be sent back on an RxRenewalResponse.

Usage of DeniedNewPrescriptionToFollow

This response type may be received in an RxRenewalResponse from a previous version of NCPDP SCRIPT and is included for backwards compatibility.

Follow Up Request Requirements

The FollowUpRequest is used to resubmit an electronic request that has not yet received a response. This element should be utilized when a system permits a user to resubmit a request. To help avoid unnecessary subsequent requests due to questions on messages status (e.g. has the receiving system confirmed receipt), applications should provide users with the ability to view the status of any outbound message.

Receivers of messages that support the FollowUpRequest element should ensure that end users are able to differentiate the most recent follow up request when multiple requests have been received. The most recent follow up request is the message that shall receive the response (Approved, Denied, etc). Customer systems should consider workflows that allow requests that will not receive a response to be noted as complete or archived to help eliminate confusion.

Veterinarian Prescriber and NonHuman Patient

Surescripts does not support the NCPDP SCRIPT elements for veterinarian prescribers or non-human patients. Messages that are sent with the new elements will be rejected.

Drug Name Best Practices

Customers should work with their drug database vendors to obtain drug database vendor’s e-prescribing preferred name; the e-prescribing preferred name should align with industry best practice standards as well as Surescripts requirements.

  • It should be noted that medications that are sent with the same drug name and form may appear slightly different among customers depending on the system’s drug compendia.

Example:

Prevacid 30 MG Oral Delayed-Release Capsule

Prevacid 30 MG DR Capsule

Sig Text Field Best Practices

  • The medication’s indication should be included within the SigText field. Sending the medication’s indication can help patients organize and better understand their medication regimen. Including the medication’s indication within the SigText field can help prevent dispensing errors and provide pharmacists and pharmacy staff additional context needed for medication therapy management and patient counseling.
    Example: "Apply a pea-sized amount sparingly four times a day for poison-ivy rash."
  • Sig information should be clinically correct.
    Example: Use "Amoxicillin 500 mg Oral Capsules"—SigText "One capsule three times a day" instead of "500 caps 3 times a day."

  • SigText should be transmitted in patient-friendly English. If the patient’s primary language is not English, the LanguageNameCode field in the patient structure should be sent with the applicable language code. Latin abbreviations should not be used. Example: Use "Take 1 tablet by mouth twice a day" instead of "1 T PO BID."

  • Sigs should be complete, properly formatted, and not repeated. See table below for example usage:

  • The instruction "take as directed" is rarely appropriate and should be avoided by prescribers. Using the phrase "take as directed" provides little information upon which the pharmacist can base their counseling of the patient.
  • In instances in which a Sig builder is utilized to provide standardized Sig options, prescriber systems should allow the end user to append free text to the standardized Sig.
  • The prescriber system should allow the prescriber to opt out of the standardized Sig and create a completely custom Sig.

Communication Numbers Requirements

When CommunicationNumbers is sent, one occurrence must be PrimaryTelephone for prescriber, supervisor, pharmacy, or facility.

For PrimaryTelephone, Surescripts validates that the number is:

  • Exactly 10 digits
  • Not all zeros or repeating numbers
  • No sequential number (1234567890, 0123456789, 9876543210)
  • Not 555 as the area code

NDC Best Practices

An NDC, by definition, is specific to a manufacturer/labeler, product, and associated packaging information. A representative NDC is an 11-digit NDC code that is intended to depict a category of medication regardless of package size and manufacturer/labeler. A representative NDC is not intended to infer specificity or preference to the imbedded manufacturer/labeler. In order to maximize the opportunity that the selected NDC exists among the various drug database vendor’s files, a representative NDC should be a nationally available product and (not be a repackaged NDC, obsolete NDC, private label NDC, unit dose NDC, or inner package/clinic package) unless it is the only NDC available identifying that category of medication. The drug description of the product should match the description of the representative NDC code value.

Surescripts will use the DEA Schedule to determine how to process the message. If the DEA Schedule is not present, Surescripts will validate NDC to determine controlled substance status. If the NDC indicates that it is a controlled substance, the message will be rejected because the DEA Schedule is required for a controlled substance. If the NDC is not found, Surescripts will pass the prescription along to the pharmacy.

Note Best Practices

EPCS (21 CFR Section 1306.11) allows the prescriber to send an electronic prescription to "cover" for his verbal order. When sending the electronic prescription this field should be populated with "Emergency Cover Prescription - Do Not Fill".

Proper use of the Note field (referred to as "free text" in the NCPDP SCRIPT Standard).

Reserve the use of the Note field for information related to, but not part of, the prescription. For example:

  • The patient's flavoring choice.
  • Multiple packaging (e.g. split up the quantity into one for school/one for home, etc.).
  • Reminder to suspend use of contraindicated medication until other drug therapy complete.

Directions for use or administration should not be split between the SigText and Note fields nor should the information in the Sig field conflict with information in the Notes field.

ProhibitRenewalRequest

The ProhibitRenewalRequest element is used to denote that an RxRenewalRequest should not be sent to the sending prescriber for this particular prescription. This is usually used when the visit is for a one time prescription (i.e., Urgency Center or Emergency Department). In addition to service level validation, the RxRenewalRequest process needs to use this flag for proper routing of the request.

If an RxChangeRequest is sent and an Approved response is given, there is no field to denote that renewals should not be sent to the sending prescriber. The pharmacy system would need to carry forward the flag from the original NewRx.

State Specific Identifiers for EPCS

Customers are responsible to meet the all applicable laws and requirements, including, but not limited to, local and state laws and regulations in which the customer’s application is deployed and used. This section gives guidance where the states have redefined how the state license number fields are used specifically for EPCS.

Identifier for Midlevel Practitioners

Some states require midlevel practitioners (i.e., advance practice nurses (APRNs), physicians assistants (PAs), etc.) with prescriptive authority to include their prescriptive authority identification number in all prescriptions they create. In addition to their prescription authorization number, midlevel practitioners are required to also include their DEA number and may be required to include the DEA number of the supervising prescriber on all prescriptions for controlled substances. When sent, the midlevel practitioner’s information should be placed in the message in the prescriber loop and the supervising physician's DEA number should be placed in the supervisor loop.

E-prescribing vendors should populate the Prescriber State License Number field with the prescription authorization number on all prescriptions (for both non-controlled and controlled substances).

Prescriber loop:

StateLicenseNumber tag in the Identification element in the prescriber section.

For APRNs, the letters "APRN" should always appear first in the field followed by the actual number.

Example: APRN123456789

DEANumber tag in the Identification element in the prescriber section.

Supervisor loop:

DEANumber tag in the Identification element in the supervisor section.
