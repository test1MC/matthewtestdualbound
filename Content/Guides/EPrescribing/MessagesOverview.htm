Messages Overview

This section provides guidelines for the data messaging interfaces between the prescriber system and the pharmacy. Some message flows include the utilization of LTPAC information. LTPAC is a variety of settings and services which help meet both the medical and non-medical needs of people with a chronic illness or disability who cannot care for themselves for long periods of time.

Notes:
  • EPCS and LTC will have specific sections below where applicable.
  • For the purposes of this Guide, when referring to LTPAC the term prescriber could be used in place of provider/prescriber/agent.

Prescriber Message Workflow

The following steps depict the basic workflow of a prescriber initiated message:

  1. Prescriber initiates a message for a patient via a prescriber system and transmits to Surescripts.
  2. Surescripts validates the message:
    • If there is an issue with the message, Surescripts returns an Error back to the prescriber system and the process ends.
  3. If no error in validation occurs, Surescripts routes the message to the appropriate pharmacy system.
  4. The pharmacy system then validates the message and responds with a synchronous Status message (Code of either 000 or 010) or an Error to Surescripts.
  5. Surescripts sends the synchronous response to the prescriber system: 
    • If either the Status message (Code 010) or Error are returned, then the process ends.
  6. If the synchronous message is a Status (Code 000) message, then the pharmacy system follows up with an asynchronous (free standing) Verify message (Code 010) or Error.
  7. When Surescripts receives either the Verify or Error, the message is validated and forwarded to the prescriber system and the workflow is complete.

Pharmacy Message Workflow

The following steps depict the basic workflow of a pharmacy initiated message:

  1. Pharmacy initiates a message for a patient via a pharmacy system and transmits to Surescripts.
  2. Surescripts validates the message:
    • If there is an issue with the message, Surescripts returns an Error back to the pharmacy system and the process ends.
  3. If no error in validation occurs, Surescripts routes the message to the appropriate prescriber system.
  4. The prescriber system then validates the message and responds with a synchronous Status message (Code of either 000 or 010) or an Error to Surescripts.
  5. Surescripts sends the synchronous response to the pharmacy system. 
    • If either the Status message (Code 010) or Error are returned, then the process ends.
  6. If the synchronous message is a Status (Code 000) message, then the prescriber system follows up with an asynchronous (free standing) Verify message (Code 010) or Error.
  7. When Surescripts receives either the Verify or Error, the message is validated and forwarded to the pharmacy system and the workflow is complete.

An asynchronous (free standing) Verify message shall only used after a Status message (Code 000).

NewRx

The NewRx message was developed using the NCPDP SCRIPT standard. A Status, Error, or Verify response is always sent after a NewRx message.

NewRx EPCS Requirements

  • For CS-II only, one dispense event is permitted per the DEA IFR. This means on a New Rx for a CS-II, the refill value must be equal to "0" (zero). Prescribers desiring to send a subsequent NewRx prescription(s) will send NewRxs with an effective date set 30 days, 60, days, 90 days, etc. in the future and the refill value on each will be set to "0" (zero).

NewRx LTC Requirements

  • An open order is defined by specific usage of the medication quantity elements.  An open order must have a medication CodeListQualifier = "QS" and use the corresponding quantity value = "0" (zero).  Open orders are dependent on regulatory expiration and audit processes in LTPAC settings and services. 
  • Open orders must not be sent for controlled substances when regulatory mandates apply.

NewRx LTC Best Practices

  • Prescriber initiated Change Process for LTC can be found in the Prescriber Initiated Change Process for LTC - Requirements/Best Practices section.
  • OrderCaptureMethod should be used in all instances of NewRxs for LTPAC settings and services.
  • Facility element information should be sent on all NewRx messages.
    • At a minimum, the Facility elements requires Identification. One or more Identification elements may be sent as needed, but at least one is required to be sent. Per NCPDP guidance in the Prescriber, Pharmacy and Facility Identifiers, the NPI is expected to be sent if one is known.
  • Patient Location fields for unit, room and bed are important to the pharmacy profile of a resident and should also be sent when known.

RxRenewal

An RxRenewalRequest is originated by the pharmacy. This message is for the purpose of requesting approval for additional refills of a prescription.

If a patient has approved refills remaining on a current prescription, no electronic message is required for a dispensing to occur.

An RxRenewalResponse is sent by the prescriber to the pharmacy in response to a request to refill a prescription. The response indicates whether the RxRenewalRequest has been Approved, Approved with Changes, Denied, or Replace. An Approved, Approved with Changes, or Replace response is an indication to the pharmacy to discontinue dispensing the original prescription and instead begin dispensing the new prescription. In this case, a CancelRx is not sent for the original prescription following the RxRenewalResponse.

The RxRenewalResponse no longer contains a MedicationPrescribed or a MedicationDispensed element, as it did in lower versions. The MedicationResponse element contains the medication information that the prescriber is approving within an RxRenewalResponse. The Approved, ApprovedWithChanges or Replace RxRenewalResponse is considered a fillable message.

RxRenewal Requirements

  • Fillable RxRenewalResponse messages for controlled substance medications must be digitally signed.
  • When sent, PharmacyRequestedRefills must not be "0" (zero). Do not send this element if you want to allow the prescriber to determine appropriate number of refills.

RxRenewal Best Practices

  • The RxRenewalRequest contains a MedicationPrescribed and a MedicationDispensed loop. The MedicationDispensed element is the required element to be used per the RxRenewalRequest schema. In order to support the translation of messages between versions, both loops should be present.

Examples:

NewRx sent by prescriber with substitutions allowed.

MedicationPrescribed: Lipitor 10 MG Oral Tablet

NumberOfRefills: 1

Pharmacy dispenses Atorvastatin 10 MG Oral Tablet

Pharmacy sends an RxRenewalRequest with the following information

MedicationPrescribed: Lipitor 10 MG Oral Tablet

NumberOfRefills: 1

MedicationDispensed: Atorvastatin 10 MG Oral Tablet

PharmacyRequestedRefills: 2

Prescriber approves (based upon MedicationDispensed) and sends an RxRenewalResponse

Response type Approved:

MedicationResponse: Atorvastatin 10 MG Oral Tablet

NumberOfRefills:2

Response type ApprovedWithChanges:

MedicationResponse: Atorvastatin 10 MG Oral Tablet

NumberOfRefills:6

Response type Replace:

MedicationResponse: Lipitor 10 MG Oral Tablet

NumberOfRefills: 2

  • If the non-EPCS authorized prescriber receives an RxRenewalRequest for a controlled substance, they respond with a Denied message to the pharmacy. Should the prescriber wish to approve the request without having the "ControlledSubstance" service level enabled, they may then continue with their non-Electronic Data Interchange (EDI) workflow as determined by their application.
  • The NDC/ProductCode in the RxRenewalRequest should not be the sole determination of the identification of the requested medication. When the incoming NDC does not match the related NewRx or cannot be found, systems should use other potential identifiers (e.g. RxNorm codes) and must always allow manual review of the drug description by the user. Therefore, all response types should be permitted (i.e. approved, approved with changes, denied, replace).

For more information on how to send a Follow Up Request, see the Follow Up Request Requirements section.

RxRenewal Prescriber Best Practices

  • Prescribers should review all requests. Prescribers should not delete a request message from a processing queue without a response being generated to the requestor.
  • Matching a request back to an original message is not required to present the RxRenewalRequest to the Prescriber. See the Message Linkage section for more details.
  • If another prescriber is responding to an RxRenewalRequest for non-controlled substances on the original prescriber's behalf (i.e., original prescriber who is out of the office), then one of the following options should be used.
    • Option A – Using PrescribingAgent – If the prescriber is out for a short time, then the prescriber field in the RxRenewalResponse should contain the prescriber's name, but the PrescribingAgent element should contain the name of the agent that is authorized to approve/deny refills for the prescriber.
    • Option B – If the original prescriber is unable to respond, the original RxRenewalRequest should be responded to with the RxRenewalResponse type of Replace. The new prescriber information should be populated in the Prescriber elements. Additionally, the new prescriber’s SPI should be sent in the From element of the message header. If needed, the original prescriber information can be sent in the FollowUpPrescriber elements.

RxRenewal Pharmacy Best Practices

  • If the prescriber has not responded within 72 hours the FollowUpRequest workflow should be utilized. See RxRenewal LTC Best Practices section for exception information.
  • A pharmacy may send an RxRenewalRequest for controlled substances regardless of whether the prescriber has the "ControlledSubstance" service level.
  • For RxRenewal routing:
    • Retain the original sending SPI for the electronically received NewRx. Evaluate this SPI (e.g., has Refill service level), and send RxRenewalRequest to this record.
    • Prescribers are allowed to have multiple SPI, it is important to ensure that when sending an RxRenewal the correct SPI is selected based on service level and location. For example confirm the Individual NPI, Address Line 1 or Address Line 2, Postal Code, Phone Fax, or Clinic Name prior to sending the message.

RxRenewal EPCS Requirements

  • For CS-II only, one dispense event is permitted per the DEA IFR. This means on the RxRenewalResponse for a CS-II the MedicationResponse, the NumberOfRefills value shall be equal to one (1) on an Approved or Approved with Changes response type.
  • RxRenewalResponse type Replace must not be sent for controlled substances when the receiver uses a lower version because of the inability to translate the DigitalSignature. Instead prescribers should deny and follow up with a new prescription.

RxRenewal LTC Best Practices

  • RxRenewal requests may be generated as an option for open orders that have hit their regulatory expiratory limit.
    • Prescribers in the LTPAC industry may not respond to RxRenewal Requests in a care setting for up to 30 days.

RxChange

An RxChangeRequest message is originated by the pharmacy to request a change to a new or fillable prescription. Response types for RxChange are Approved, ApprovedWithChanges, Denied, and Validated.

An RxChangeRequest may be sent with the following MessageRequest codes:

  • “G”, may be used to request a prescriber allow the dispensing of a generic medication.
  • “P”, may be used to request a prescriber review the drug requested and obtain a prior authorization from the payer for the existing prescription.
  • “T”, may be used to request a prescriber authorize a therapy change to the original prescription.
  • “D”, may be used to request a prescriber authorize a change to an alternative medication that will have fewer, or no, adverse effects than the original prescription.
  • “S”, may be used to request a prescriber clarify the original prescription.
  • “OS”, may be used to request a prescriber authorize a change to the original prescription due to the pharmacy being out of stock of the requested medication.

  • “U”, may be used to request prescriber authorization information, such as confirming their DEA number or enrollment with the prescription benefit plan. The “Validated” RxChangeResponse type is only used when responding to this type of request.

RxChange Requirements

  • An RxChangeResponse type of ApprovedWithChanges cannot be used in response to an RxChangeRequest with a MessageRequestCode of "P" (Prior Authorization).
  • MedicationPrescribed is required when an RxChange response is Approved or ApprovedWithChanges.

RxChange Best Practices

  • At least one loop of the MedicationRequested loop should be sent, unless the use cases are for PriorAuthorization (P) or PrescriberAuthorization (U).

  • The MedicationRequested loop utilizes discrete fields to provide the requested changes and the Note field to provide additional information that is not captured discretely for that loop.

  • Approved responses are utilized when one of the MedicationRequested loops is echoed in the MedicationPrescribed loop of the RxChangeResponse with no changes to the requested elements.

  • Approved with Changes responses are used when there are changes other than what was requested. The Approved with Changes response is considered a new fillable prescription and indicates to the pharmacy that the in process prescription should no longer be dispensed and should be cancelled.

  • Prescriber vendors utilizing Surescripts Prior Authorization should optimize workflows by automatically triggering a PAInitiationRequest when they receive an RxChangeRequest for prior authorization and an RxChangeResponse upon receipt of a prior authorization approval.

RxChange LTC Best Practices

  • LTPAC settings and services do not always have prescribers available to approve or deny changes when initiated by a pharmacy which can make it difficult to support the RxChange messages. In many cases, facilities and pharmacies generally have contracts that apply to generic and therapeutic substitutions. RxFill is used in many cases to communicate when substitutions are made. If there is a pharmacy requested change to a prescribed therapy that is beyond a substitution and RxChange is not supported, an alternative method of communication is recommended.

  • The Prescriber Initiated Change Process is used for prescribers to alter existing therapies and utilizes a NewRx > CancelRx > NewRx process with appropriate message linking and MessageRequestCode field indicators. This process is outlined in the CancelRx section. Appropriate message linking and field population as outlined in the CancelRx section.

CancelRx and CancelRx Response

The CancelRx is used to notify the pharmacy that a previously sent prescription should be cancelled and not filled. The message is originated by the prescriber system as a CancelRx.

The CancelRxResponse is sent from the pharmacy to the prescriber system in response to a CancelRx.

Prescriber Initiated Change Process for LTC - Requirements/Best Practices

  • R.208: In LTPAC prescriber systems, when making a prescription change using a NewRx, a MessageRequestCode and OrderCaptureMethod shall be used to identify the type of change and the source of the data. These fields shall be present in both the CancelRx and the subsequent NewRx.
  • The ChangeOfPrescriptionStatusFlag should contain the appropriate code for canceling a medication that has not been dispensed or discontinuing an active order.
  • Scenarios for Prescriber-Initiated Changes are as follows:
    • A data entry error is corrected by a prescriber or an agent of a prescriber and a CancelRx is sent with a subsequent NewRx.
    • A prescriber alters a course of therapy after making a clinical decision regarding a medication and sends the pharmacy a CancelRx followed by a subsequent NewRx.
  • CancelRx should contain a CodeListQualifier value of "QS" and Quantity of "0" when discontinuing an open order.
  • CancelRx and CancelRxResponse should also be accompanied by a ChangeOfPrescriptionStatusFlag. This is used on the CancelRx to indicate prescriber knowledge of dispensing/administering activity, and on the CancelRxResponse to indicate the pharmacy acknowledgement of dispensing process. 

Values:

C = Use for canceling a prescription prior to a dispensing (administration)

D = Used for canceling a prescription that has been dispensed (administered)

Fill Messages

RxFill messages are used to provide prescribers with information regarding dispensing activity for a prescription. Additionally, prescribers may select the types of notifications they desire over the life of the prescription. 

To optimize use of the RxFill transaction, both prescriber and pharmacy systems utilizing the RxFill messages must support the RxFillIndicator element and the RxFillIndicatorChange message. For prescriber systems, support of the RxFillIndicator element includes setting the desired value in prescriber initiated messages such as NewRx, RxRenewalResponse, RxChangeResponse, and Resupply. For pharmacy systems, this includes sending fill status notifications based on the indications selected by the prescriber as well as adding or altering preferred notification types when the RxFillIndicatorChange is utilized. 

Pharmacy systems must be able to support the following fill status types: Dispensed, PartiallyDispensed, and NotDispensed. Support for sending a fill status of Transferred is required when the pharmacy system also supports the RxTransfer message.  While the use of Reason Codes or Notes is optional to send in RxFill messages, pharmacy systems are strongly encouraged to support the inclusion of these additional elements. 

Fill Status Workflow and Scenarios

Prescribers initiate the process of requesting fill status messages through the use of RxFillIndicator in prescriber initiated messages. The pharmacy then tracks the desired indications. The prescribers may add or alter selected fill status options when sending subsequent prescriber initiated messages. Additionally, prescribers have the ability to send an RxFillIndicatorChange message to request alterations to preferred fill status notifications.

The following scenarios are examples of how the RxFillIndicator and RxFillIndicatorChange message can be used.

Scenario 1 – Prescriber opts into receiving RxFill notifications and then alters preferred selection:

  1. Prescriber initiated message sent with RxFillIndicator signifying “All Fill Statuses” to a pharmacy that is enabled for RxFill.
  2. Pharmacy processes the message and tracks the preferred fill status options.
  3. Prescriber chooses to alter selected fill status options to “Not Dispensed” using the RxFillIndicatorChange message.
  4. Pharmacy updates the requested types and sends fill status only when prescription is not dispensed.
    • This RxFill message type may be sent, for example, when a patient does not pick up an automatic refill and the prescription is returned to stock.

Scenario 2 – Prescriber initiates the process to indicate preference for PartialDispense and NotDispensed only, and then chooses to not send any RxFillIndicator on an RxRenewalResponse:

  1. Prescriber sends NewRx with RxFillindicator signifying “Partially Dispensed And Not Dispensed” only.
  2. Pharmacy processes the message and tracks the preferred fill status options.
  3. Pharmacy sends RxFill messages only when the selected dispensing activity occurs.
    • This RxFill message type may be sent, for example, when a regulatory limitation causes the patient to only be dispensed a partial fill.
  4. Pharmacy sends RxRenewalRequest.
  5. Prescriber sends an RxRenewalResponse with no RxFillIndicator. 
  6. Pharmacy system processes the message. In this case, no fill status notifications are sent.

A fillable message indicates the preferred options going forward. In Scenario 2, for example, no RxFillIndicator is sent in the RxRenewalResponse, so the pharmacy would send no RxFill messages, even though the original NewRx had indicated a fill status preference.

Scenario 3 – Prescriber initiates the process to indicate a profile only medication:

  1. Prescriber sends NewRx with no RxFillIndicator and additionally indicates “DoNotFill”.
  2. Pharmacy processes the message and no fill status messages are sent to the prescribing system.

As per Surescripts Directories requirements, RxFillIndicatorChange messages will only be sent to pharmacies that have the service level enabled to allow them to receive that message type. 

RxFill Requirements

  • One medication loop must contain WrittenDate. If the MedicationPrescribed loop is not present, then the MedicationDispensed loop must provide WrittenDate.
  • When the status of the message is Dispensed or Partially Dispensed, then MedicationDispensed must be sent. When the status of the message is Not Dispensed or Transferred, either the MedicationPrescribed or MedicationDispensed must be sent. For any status, both MedicationPrescribed and MedicationDispensed may be sent.

Fill Status Best Practices

In the context of an RxFill message, ‘dispensed’ indicates the medication is no longer in the possession of the pharmacy and has been handed, shipped, or delivered to the patient (or the patient’s caregiver/representative). If the medication is still located in the pharmacy, it is not yet ‘dispensed’.

  • Fillable messages and Resupply should contain the RxFillIndicator if RxFill messages are desired. 
  • In order to receive fill status notifications the prescriber or prescriber system sets the preferred fill status options on each fillable message and Resupply.
  • Send Facility NPI in the Identification element when known.

LTC Best Practices

  • RxFill messages are designed to be sent based on the prescriber indication. When a prescription is sent to a pharmacy via non-electronic means (e.g. fax), the pharmacy will not have an indication of what types of fill status notifications the prescriber desires. In the LTPAC care setting, there is still a need to receive an RxFill message for prescriptions sent via a non-electronic method. Partnership agreements often allow and/or require RxFill messages to be sent, regardless of the method in which the pharmacy received the prescription. 
  • Resupply messages include the ability to alter selected RxFillIndicator preferences.
    • If no RxFillIndicator element is indicated in the Resupply message, no change would be made to previous preferences. 

Acknowledgement Messages

This section contains acknowledgement details, workflows, and scenarios.

Status

Status is used to indicate to the sender that the message was accepted for processing. A Status is always a synchronous response. Status 010 indicates that message processing is complete and was successful, Status 000 indicates that the final response is pending.

A Status 000 indicates the process is not complete. A subsequent acknowledgement message should be expected.

Status Best Practice

DescriptionCode is an optional element that contains codes suitable for use on error messages only. DescriptionCode element should not be used on a Status message.

Verify

This NCPDP SCRIPT message is returned only as an asynchronous response as a follow up to previous Status 000. The Verify message confirms that the receiver successfully received and accepted the message.

A Verify indicates the process is complete. A subsequent acknowledgement message should not be expected.

Error

This message indicates that an error has occurred. An error can be generated when there is a communication problem or when an error occurs during the processing of the message.

Error Best Practice

In the case where the customer’s time limitation threshold has been exceeded, and a syntactically valid message has been received, an error is an incorrect response.

Acknowledgement Message Workflow Scenarios

The following workflows depict various scenarios where Verify, Error, and Status are sent in response to a NewRx request. Note that for all of these scenarios, NewRx could be any of the following messages: RxRenewalRequest, RxRenewalResponse, RxFill, RxChangeRequest, RxChangeResponse, CancelRx, and CancelRxResponse.

Customers must support all messages as described in the Acknowledgement Message Table. Every message workflow ends with Status (Code 010), Verify, or Error.

Scenario 1 – Message errors at Surescripts

1a). A NewRx is sent.

1b). Surescripts receives the NewRx.

1c). Surescripts does not recognize the format of the message, or errors are contained within the message, and returns an Error or error text to the sending customer.

Scenario 2 – Message is acknowledged successfully by recipient

2a). A NewRx is sent.

2b). Surescripts forwards the message to the receiving customer.

2c). Receiving customer validates the message and returns a Status (Code 010) to Surescripts.  This indicates the message was accepted by the ultimate receiver.

2d). Surescripts forwards the Status (Code 010) back to the sending customer.

Scenario 3 – Message errors at Receiver

3a). A NewRx is sent.

3b). Surescripts forwards the message to the receiving customer.

3c). Receiving customer either does not recognize the format of the message, or errors are contained within the message, and returns an Error message or error text to Surescripts.

3d). Surescripts sends an Error to the sending customer.

Scenario 4 – Status 000 followed by an Error

4a). A NewRx is sent.

4b). Surescripts forwards the message to the receiving customer.

4c). Receiving customer validates the message and returns a pending Status (Code 000) to Surescripts. This indicates that the message has been received but processing is not yet complete and the message has not reached its ultimate destination.

4d). Surescripts forwards the Status (Code 000) back to the sending customer.

4e). Receiving customer encounters an error when processing the message and initiates a new Error (sometimes referred to as a "Free-Standing Error").

Surescripts may convert the message to fax and avoid sending the error back to the sending customer. See the Faxing section for more information.

4f). Surescripts forwards the Error to the sending customer.

4g). Sending customer acknowledges the Error by sending back a verified Status (Code 010) to Surescripts.

4h). Surescripts forwards the verified Status (Code 010) to the receiving customer.

Scenario 5 – Status 000 followed by a Verify

5a). A NewRx is sent to Surescripts.

5b). Surescripts forwards the message to the receiving customer.

5c). Receiving customer validates the message and sends back a Status (Code 000) to Surescripts. This indicates that the message has been received, but processing is not yet complete and the message has not reached its ultimate destination.

5d). Surescripts forwards the Status (Code 000) back to the sending customer.

5e). Receiving customer delivers the NewRx to the end point and initiates a new Verify (sometimes referred to as a "Free-Standing Verify").

5f). Surescripts forwards the Verify to the sending customer.

5g). Sending customer acknowledges the Verify by sending back a Status (Code 010) to Surescripts. Code 010 indicates it was accepted by the ultimate receiver.

5h). Surescripts forwards the Status (Code 010) to the receiving customer.

Scenario 6 – Connectivity Error with receiving customer

6a). A NewRx is sent to Surescripts.

6b). Surescripts attempts to forward the message to the receiving customer, but is unable to connect and receives an HTTP error or the connection times out after 24 seconds.

6c). Surescripts sends back a Status (Code 000) to the sending customer. This indicates that the message has been received, but processing is not yet complete and the message has not reached its ultimate destination.

6d) Surescripts makes up to 3 additional delivery attempts with a delay of at least 1 minute, 5 minutes and then 10 minutes between each attempt.

6e). If one of the delivery attempts is successful and the receiving customer returns a Status (000), Surescripts will audit the response, but it will not deliver that recent Status (000) to the sending customer as they have already received one from Surescripts in step 6c.  Final status will be sent to the sending customer once the recipient sends a free standing Error or Verify as in steps 4e and 5e above.

6f). If one of the delivery attempts is successful and the receiving customer returns a Status (010), Surescripts will convert the Status (010) into a Free Standing Verify and deliver the Verify to the sending customer.

6g). Sending customer acknowledges the Verify by sending back a Status (Code 010) to Surescripts. Code 010 indicates it was accepted by the ultimate receiver.

6h). If one of the delivery attempts is successful and the receiving customer returns an Error, Surescripts will forward the Error to the sending customer.

Surescripts may convert the message to fax and avoid sending the error back to the sending customer.  See the Faxing section for more information.

6i). Sending customer acknowledges the Error by sending back a Status (Code 010) to Surescripts. Code 010 indicates it was accepted by the ultimate receiver.

6j). If none of the delivery attempts are successful Surescripts will create send a Free Standing Error to the sending customer.

Surescripts may convert the message to fax and avoid sending the error back to the sending customer.  See the Faxing section for more information.

6k). Sending customer acknowledges the Error by sending back a Status (Code 010) to Surescripts. Code 010 indicates it was accepted by the ultimate receiver.

Faxing

This service is optional and creates an alternate route for transactions if electronic communication is disrupted. This service delivers certain messages via facsimile (FAX) when they cannot be delivered electronically due to temporary/transient transmission failures, unless prohibited by state law. Both sender and receiver must have this feature enabled to utilize faxing services.

Notes:
  • Fillable messages for controlled substances will not be transmitted by fax.
  • If fillable fax messages are prohibited by state, messages sent via fax are for informational purposes only.

For faxing, consider the following:

  • Current message types that are supported through faxing: NewRx, RxRenewal (Request/Response), and CancelRx (Request only).
  • Pharmacies may send an RxRenewalRequest for controlled substance that can fail over to a fax message when a connectivity issue occurs.
  • If a prescription for a controlled substance is sent, and a connectivity issue is encountered, an Error will be returned to the sender instead of failing over to a fax message.
  • Surescripts makes efforts to conform to the applicable state regulatory needs related to e-prescribing; however, it is ultimately the responsibility of the sender and receiver to fulfill compliance requirements.
  • Information only faxes – Some states permit information only faxes for non-controlled substances. These fax templates meet the state requirements for transmitting message details by fax, but provide further instruction for the pharmacy to validate the prescription.

Message Delivered by Fax

The following steps occur when a message fails over to fax:

  1. Electronic message is initiated and received by Surescripts.
  2. Connectivity issue from the receiver is encountered by Surescripts.
  3. Surescripts creates a Status (Code 000) message and returns to sender. 
  4. Surescripts fails the message for delivery via fax after determining that the message meets all required criteria.
  5. The fax delivery process is monitored for success.
  6. When confirmation is received for successful delivery of fax, a Verify (Code 010) status update is returned to the sender to notify that the message was delivered by fax.
  7. If a timeout occurs, Surescripts will return an error to the sender and a message may still be delivered via fax. Duplicate messages should not be sent without confirming that the original message was received.

Faxable Error Messages

Surescripts can attempt to deliver the original message by fax when receiver respond with certain error code. The following steps summarize the process:

  1. Surescripts receives electronic message and attempts to deliver to the receiver.
  2. Surescripts receives synchronous error message with code 600, 601 or 602 or a status 000 followed by asynchronous error message with code 600, 601 or 602. Codes that are faxable include: 600, 600-008, 601, 602, 602-008. Any other combination is not faxable.  
  3. Surescripts returns status 000 and fails the message over for delivery via fax after determining that the message meets all required criteria.
  4. The fax delivery process is monitored for success.
  5. When successful, a Verify (Code 010) status update is returned to the sender to notify that the message was delivered by fax.
  6. If a timeout occurs, Surescripts will return an error to the sender.

Fax Portal Pharmacy Configurations

This faxing configuration is at a portal level for pharmacies that do not receive messages electronically. A pharmacy that does not transmit electronic messages will only receive NewRx. This type of pharmacy will be identified through pharmacy directory services.
