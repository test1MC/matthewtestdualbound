Sending SMTPXDR Messages

Sending a message refers to the customer XDR interface initiating an XDR transaction to submit a Provide and Register Document Set-b, ITI-41, transaction to Surescripts. The effect of this is to send a message from the customer software to a recipient through the Surescripts network.

In terms of [ITI TF-1] section 15.1, the customer XDR interface is the Document Source and Surescripts is the Document Recipient.

To address messages:

  • The customer XDR interface must include the direct:addressBlock SOAP header defined in [XDR for Direct].
  • Within the address block, one direct:from element must be present and contain the Direct address of the user (or role) that sent the message. The address must be part of the domain assigned to the customer on the network.
  • At least one recipient address must be included in the direct:to element. Surescripts allows multiple direct:to elements.

The customer XDR interface should include a text/plain document with a classCode of 56444-3 (Healthcare Communication). This document will be the body text of the email message when Surescripts translates the XDR. If the XDR message does not contain this document, Surescripts will include generic text as the message body.

The customer XDR interface may generate metadata to conform to either Full XDS Metadata or Minimal Metadata, as defined by [XDR for Direct] section 6.1.

If the receiving system is able to accept the message for delivery to the Surescripts network, it will return a success response to the SOAP call. If the gateway is unable to accept the message, it will return an error response to the SOAP call, with a RegistryErrorList entity in the RegistryResponse entity.

If the customer system is unable to connect to Surescripts, and if the message that a user is sending is able to be queued for later delivery retry, Surescripts suggests several retries over a period of three (3) hours to minimize the effect system maintenance or outages has on users.
