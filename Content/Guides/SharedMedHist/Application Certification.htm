Application Certification Requirements Summary

Sec. 2.1.5 Medication History Prescriber Vendor Requirements

1). The application shall*106M-011 allow the end user to change a patient's consent to obtain medication history between, at a minimum, the yes and no consent values.

2). The application shall*106M-012 display a disclaimer regarding use of medication history retrieved via Surescripts.

a). Sample Disclaimer: Certain information may not be available or accurate in this report, including items that the patient asked not be disclosed due to patient privacy concerns, over-the-counter medications, low cost prescriptions, prescriptions paid for by the patient or non-participating sources, or errors in insurance claims information. The provider should independently verify medication history with the patient.

3). When a data element from the medication history response is displayed, the application shall*106M-014 have the ability to display all data received for that data element.

4). If the application displays medication history from sources other than Surescripts, the application shall*106M-020 identify which medication history was retrieved via Surescripts.

5). The application shall*106M-018 have the ability to display all medication history Drug/MedicationDispensed loops retrieved, unless prohibited by federal, state or local laws.

6). The application shall*106M-019 provide access to all history data available for a specific request. When a medication history response indicates more medication history is available, the application shall*106M-026 either systematically retrieve all medication history available or notify the end user more medication history is available.

7). For PBM/Payer claim data, the participant shall*106M-016 retrieve medication history for eligible coverages only.

Note: For pharmacy fill data, the participant may retrieve medication history regardless of patient eligibility.

8). For PBM/Payer claim data, the participant shall*106M-017 retrieve medication history for all eligible coverages.

9). The claim medication history request for a patient shall*106M-013 be tied to the most recent eligibility response for that patient. This is accomplished by using the ISA control number (ISA-13) from the 271 Eligibility response in the UIB-030-02 <RelatesToMessageID> of the RXHREQ Medication History Request.

2.1.6 Medication History PBM/Payer Requirements

Drug segments shall*106M-021 be sent in order of most recent history first according to fill date.

Sec. 3.4.3.1 Numeric Representation

The decimal point is represented by a period and shall*106M-001 be used as follows:

  • only when there are significant digits to the right of the decimal
  • when there is a digit before and after the decimal point
  • not with whole numbers
  • not to be counted towards the length total of a data element (only for EDIFACT)

Sec. 3.4.3.3 EDIFACT Trimming

All non-essential characters shall*106M-002 be suppressed from the message where permissible (see section "EDIFACT Representation" above). Non-essential characters include leading spaces, trailing spaces, leading zeroes, and trailing zeroes where permissible (see section"Numeric Representation" above).

The length of SCRIPT message can be optimized in several ways.

  • Empty segments should not be transmitted.
  • Depending on the trading partner and the message, not all data elements will be utilized. All data elements within a segment after the last data element used may be dropped.

Although SCRIPT messages can be trimmed in several ways, software systems shall*106M-003 not assume that a trading partner would always trim their messages. The systems shall*106M-004 be capable of properly interpreting a full-length message. If a data element is unused, it may be omitted. However, the data element separator character must be used to "hold the place" of the data element. SCRIPT messages do not contain field identifiers; therefore, all data elements (up to the segment trim point) must be accounted for with separator characters.

Sec. 3.5 XML Empty Tags

Participant shall*106M-005 not send empty elements within XML messaging with exception of the RxHistoryResponse-<Approved> element.

All non-essential characters shall*106M-002 be suppressed from the message where permissible (see Section 3.4.3 EDIFACT Representation above). Non-essential characters include leading spaces, trailing spaces, leading zeroes, and trailing zeroes where permissible (see Section 3.4.3.1 Numeric Representation above).

Sec. 3.8 Handling Optional Fields

The participant shall*106M-007 be able to process any valid incoming transaction request or response, including syntactically valid maximum and minimum population messages. Optional data elements (and values therein) shall*106M-008 not cause message failure.

Sec. 5.3 Error Message
For PBM/Payers, an asynchronous error message (also called Free Standing Error message) is sent in response to an invalid RXHRES message. The asynchronous error message is a separate message type and shall*106M-022 be responded to with a status of 010. An Error message shall*106M-023 never be responded to with an Error message; this could create an endless loop.

The self-contained full drug name, strength, and form.
