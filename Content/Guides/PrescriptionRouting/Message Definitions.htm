Message Definitions

This section provides guidelines for the data messaging interfaces between the prescriber using a prescriber system and the pharmacy. For LTPAC application, please refer to Sec. 1.2.1 Terms Used for LTPAC in This Guide.

NewRx - New Prescription Request

The NewRx messages to be transmitted have been developed using the NCPDP SCRIPT standard. A Status, Error, or Verify response is always sent after a NewRx message.

NewRx Message Flow

The following steps depict the NewRx message flow:

  1. Prescriber initiates a new prescription for a patient via a prescriber system.
  2. Prescriber system successfully transmits the new prescription to Surescripts.
  3. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  4. Surescripts uses the Recipient ID and routes the message appropriately.
  5. Pharmacy validates the prescription request and responds with a Status, Error (if errors are found), or Verify message.
    1. If a Status 000 is sent in step 5, the pharmacy would send a Verify.
  6. Surescripts transmits the Status, Error, or Verify back to the sender.

NewRx - Prescriber Best Practices

  1. For NewRx after a Denied NewRx to Follow (DNTF) response, the prescriber application should show the Refill/Renewal denied NewRx detail on the NewRx screen, when applicable, to streamline the prescription writing process.
  2. The prescriber should use the notes field to indicate on both prescriptions (mail and retail) that this prescription has also been sent to retail/mail.

NewRx - Pharmacy Best Practices

When receiving a NewRx, the pharmacy application should display the following in its entirety to the pharmacist:

 

  • Prior Authorization Number and Prior Authorization Status
  • Observation Segment: This information (Weight, Height) is especially important for infused, injected, oncology, and pediatric medications.

RxRenewalRequest(FKA Refill)

A RefillRequest message is originated by the pharmacy. This message is for the purpose of requesting approval for additional refills of a prescription once the original number of refills has been dispensed.

If a patient has approved refills remaining on a current prescription, no electronic message is required for a dispensing to occur.

A RefillResponse is sent by the prescriber to the pharmacy in response to a request to refill a prescription. The response indicates whether the RefillRequest has been accepted or denied.

RxRenewal Total Number of Dispensings

In the NewRx, the total number of dispensings is 1 + the refill quantity, however, in the refill renewal response the total number of dispensings is only the refill quantity. The application shall 106R-053 make this distinction clear to the user during the approval process.

RxRenewalMessage Flow

The following steps depict the Refill Renewal message flow:

  1. The Refill Renewal message is successfully transmitted from the pharmacy system to Surescripts.
  2. Surescripts validates the message and utilizes the Recipient ID to route the message to the appropriate prescriber via their prescriber system. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  3. Prescriber system validates the RefillRequest and returns a Status, Error (if errors are found), or Verify message.
    1. If a Status 000 is sent in step 3, the prescriber system would send a Verify.
  4. Surescripts transmits the Status, Error, or Verify message back to the pharmacy system.
  5. Prescriber system processes the RefillRequest and returns the prescriber's RefillResponse message.
  6. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the prescriber system.
  7. Surescripts transmits the RefillResponse message back to the pharmacy system.

  8. Pharmacy system validates the RefillResponse and returns a Status, Error (if errors are found), or Verify message.

    1. If a Status 000 is sent in step 8, the pharmacy system would send a Verify.
  9. Surescripts transmits the Status, Error, or Verify back to the prescriber system.

RefillResponse Handling

General RefillResponse

On a refill response, if the pharmacy needs to be changed, the prescriber shall R6.182 send a “Denied” Response and send a new NewRx to the desired pharmacy.

An automatic response for RefillRequest shall R6.178 never be sent back to the pharmacy. User review and approval/denial of the RefillResponse shall R6.179 is always be required. Note: the exception to this requirement is for a participant that does not support CP/SP. When possible use Approve or Approve with Changes rather than Replace.

Controlled Substance RefillResponse

If the non-EPCS authorized prescriber receives a RefillRequest for a controlled substance, they should respond with a Denied message to the pharmacy. Should the prescriber wish to approve the request, they may continue with their non-EDI workflow as determined by their application.

Refill Renewal Best Practices

In SCRIPT 10.8 the mandatory drug loop was changed from “prescribed” to “dispensed”. Since we are moving toward a mandatory segment, we need to require both in the transition period. The reason for sending the prescribed is to assist the prescriber in matching the medication to what was sent on the original NewRx.

On the RefillRequest the pharmacy should use both the prescribed and dispensed medication loops to indicate the number of refills requested.

Surescripts requires that the medication prescribed and medication dispensed loops be included on the RefillRequest. The medication prescribed loop from the pharmacy should not contain the literal prescribed medication information that was provided by the prescriber on a NewRx, but instead should include the pharmacist’s interpretation of the medication ordered by the prescriber. The medication dispensed is the medication that the pharmacy dispensed to the patient. The pharmacy is requesting that the medication listed in the dispensed loop be renewed. The prescriber system should indicate that the dispensed medication is the item being requested for renewal. The medication prescribed information is displayed to the prescriber for cross-reference only.

As stated in the RES segment definition, only certain fields can be changed on a RefillResponse for an Approve. Refer to RES-010, for details.

If the RefillResponse is for a controlled substance, the prescriber is required to sign the medication prescribed information. Even though the medication prescribed is sent for cross-reference on the RefillRequest, it becomes what the prescriber is approving on the RefillResponse. Therefore, the prescriber is actually approving both what was prescribed and dispensed. The pharmacy can still substitute and dispense what was dispensed before.

Example:

NewRx sent by prescriber with substitutions allowed.

Medication Prescribed: LIPITOR 10MG TABLET
Refill Qualifier: R
Refill Value: 1

Pharmacy dispenses ATORVASTATIN 10MG TABLET

Medication Prescribed: LIPITOR 10MG TABLET (Name can be changed to match what is in pharmacy’s database for this medication.)
Refill Qualifier: P
Refill Value: 2

Medication Dispensed: ATORVASTATIN10MG TABLET

Refill Qualifier: P
Refill Value: 2

Prescriber approves and sends Refill Renewal Response

Medication Prescribed: LIPITOR 10MG TABLET
Refill Qualifier: A
Refill Value: 2

Medication Dispensed: ATORVASTATIN10MG TABLET

Refill Qualifier: A
Refill Value: 2

RxRenewal Prescriber Best Practices

  1. Prescriber should respond to all requests received within 48 hours. Prescribers should not delete a request message from a processing queue without a response being generated to the requestor.
  2. If another prescriber is responding to a Refill Renewal Request for non-controlled substances on the original prescriber's behalf (i.e. original prescriber who is out of the office), then one of the following options should be used.

    Option A – Using Prescribing Agent – If the prescriber is out for a short time, then the prescriber field in the Refill Renewal Response should contain the prescriber's name, but the Prescribing Agent field should contain the name of the prescriber that is authorized to approve/deny refills for the prescriber.

    Option B – If the prescriber is out for a long time, the original Refill Renewal Request should be denied using the DeniedNewRxToFollow (DNTF) status type and followed by a NewRx. The DNTF Refill Renewal Response should contain the original prescriber's name, and the NewRx should be from the new prescriber.

  3. If the prescriber order number cannot be used to match a NewRx to a Refill Renewal Request; the following data in the prescribed drug, prescriber, and pharmacy segments in the Refill Renewal Request should be used to link the Refill Renewal Request to the original NewRx:

    • RelatesToMessageID

    • Patient matching: name, date of birth, gender

    • Prescribed Drug – one or more of these data elements in the prescribed drug segment: Drug name, dose form, strength, NDC, FMT coded quantity

    • Prescriber – One or more of these elements: SPI, NPI, name, communication number

    • Pharmacy – One or more of these elements: NCPDP ID, NPI, Pharmacy Name, Communication Number

  4. Many of our vendors have found it helpful to couple the Refill Renewal Request/Refill Renewal Response/Denied New Prescription To Follow processes together, so they do not send the Refill Renewal Response until the NewRx has been completed. This helps keep the messages tied together and prevents issues with the pharmacy receiving the Refill Renewal Response, but not the NewRx.
  5. When a prescriber leaves a practice the prescriber should be disabled in the directory from that practice as soon as they leave to prevent renewals from being sent and approved for a prescriber no longer practicing at that location.

Refill Renewal Pharmacy Best Practices

  1. If the prescriber has not responded within 48 hours, another message should not be sent. Instead, the pharmacist should call the prescriber.
  2. A pharmacy may send a Refill Renewal Request for controlled substances regardless of whether the prescriber is set up for the EPCS service.
  3. The pharmacy application should only allow a Refill Renewal Request to be sent when there are no more refills on the original prescription to fill or if the prescription is expired.
  4. When the application receives the following data points, the screen should also display:

    • Prior Authorization Number and Prior Authorization Status

    • Observation Segment: This information (Weight, Height) is especially important for infused, injected, oncology, and pediatric medications

  5. Renewal routing:
    1. Retain the original sending SPI for electronically received NewRx. Evaluate this SPI (e.g., has Refill Renewal Service Level), and send RefillRenewalRequest to this record.
    2. Instead of selecting the first SPI in the file where the SPI has renewals enabled, evaluate all of the provider's SPIs where renewals are enabled to establish confidence to send the electronic request. Take into account the individual provider and location context necessary to find a prescriber within the appropriate organizational context (e.g., find the Individual [NPI or DEA or Name] at a Location [Address Line 1or Address Line 2 or zip code or phone or fax or clinic name], and Refill Renewal Service level).
    3. If more than one SPI is found with Refill Renewal Service level, it is recommended that additional refinement be performed based on information from the new prescription, or provide options for manual review.
    4. Pharmacies should send RefillRenewal Request via a non-EDI method (i.e. fax, call) when prescriber location is not found.

RxChange

RxChange General Message Flow

The following steps depict the RxChange message flow:

  1. A NewRx is processed by the pharmacy and the pharmacy identifies that a change to the prescription may be appropriate.
  2. Pharmacy submits the RxChangeRequest via Surescripts to the original prescriber.
  3. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  4. Surescripts successfully transmits the RxChangeRequest to the prescriber via the prescriber system.
  5. Prescriber system validates the RxChangeRequest and returns a Status or Error (if errors are found) message.
    1. If a Status 000 is sent in step 5, the prescriber system would send a Verify.
  6. Surescripts transmits the Status, Error, or Verify message back to the pharmacy system.
  7. Prescriber system processes the RxChangeRequest and returns the prescriber's Change Response message.
  8. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the prescriber system.
  9. Surescripts transmits the RxChangeResponse message back to the pharmacy system.
  10. Pharmacy system validates the RxChangeResponse and returns a Status or Error (if errors are found) message.
    1. If a Status 000 is sent in step 11, the pharmacy system would send a Verify.
  11. Surescripts transmits the Status, Error, or Verify back to the prescriber system.

RxChange Best Practices

  1. Notes specific to the reason for the request should go in the prescribed drug segment as defined in the use cases above. Notes specific to all drugs requested should go on the first requested drug segment.
  2. The COO Benefit segment should be included so the prescriber knows which payer the pharmacist is using to determine coverage/pricing.
  3. If the Rx Change for Therapeutic Interchange or Generic Substitution is approved, this replaces the original Rx. The prescriber should create a new PON (prescriber order number); the old one becomes inactive for this approval. If the Rx Change for Prior Authorization is approved, the original prescription can still be used.
  4. A (RxChange) denial does not make the prescription invalid or stop therapy. If the prescriber wishes to stop therapy, the prescriber should send a cancel message for the original NewRx.
  5. An Approved with Changes Response (RES-010 = C) should not be used for Prior Authorization request; it has been removed in a future version of SCRIPT.
  6. Even though the request segment is optional, it should be sent to identify the type of request and it also should be returned on the response.
  7. A pharmacy may send an RxChangeRequest for controlled substances (EPCS) regardless of whether the prescriber is setup for the EPCS service. For approvals, when received, if the prescriber is not enabled for the EPCS service, the prescriber should deny the electronic request with status type "denied" and indicate in the free text field that the denial will be followed by a written or telephoned prescription order. If the prescriber is set up for the EPCS service they can approve the change request as stated in the requirements. Denials should occur as stated above in Section 5.3 for the specific use case that applies.
  8. The free text field (RES-040) should be accessible for entry and accessible for prescriber viewing (added per advisory group notes).
  9. If the prescriber approves the change response it should be treated just like a new prescription. With a change response there are no limits or requirements around what can or cannot change in the message like there are on a renewal response. The pharmacy system should absorb the response data like they would on a new prescription in order to capture all the necessary information the prescriber sent.

RxFill

The RxFill message is originated by the pharmacy. This message notifies the prescriber about the status of a prescription - either new, renewal or resupply. RxFill messages should be viewed by the prescriber as individual snapshots in time regarding a prescription; not a final or complete status. The RxFill message can be used in three instances:

  • to notify of a dispensed prescription (the patient picked up the medication or the medication will be delivered to the specified LTC organization),

  • to notify of a partially dispensed prescription (patient picked up part of the medication or part of the medication will be delivered to the specified LTC organization), and

  • to notify of a prescription never dispensed, (patient did not pick up the medication or the medication will not be dispensed at the present time).

The differentiation of the three cases is with the use of the FillStatus XML tags.

RxFill Assumptions

  • The implementation of RxFill messaging will be a transition over a significant period of time due to the requirements for coding and rollout of pharmacy and prescriber systems to utilize RxFill.
  • Since RxFill is not a mandatory transaction, utilization will likely not achieve 100% participation.
  • Participation in RxFill usage is voluntary for both pharmacies and prescribers.
  • RxFill messages may be sent for prescriptions received by the pharmacy in any form (paper, fax, phone, or electronic) when the prescriber associated with the prescription is accessible via the electronic network.
  • Since all pharmacies and prescriber practices may not have all the technology required to send all types of RxFill messages, some percentage of providers will not participate in RxFill message tracking.
  • Because of the variation in participation, capability, and timing by pharmacies in the utilization of RxFill messages, the prescriber should be careful in the analysis and assumptions based on their usage.

RxFill for LTC Assumptions

  • Implementation of RxFill is standard in LTC pharmacy/organization communication and is used to inform the LTC organization of medication review and pharmacy approval for dispensing. In addition, this message indicates that a medication order has/has not been dispensed by the pharmacy.
  • Where RxFill is not a mandatory transaction, utilization in LTC is likely to still be 100% participation as most trading partner agreements consider this transaction mandatory.
  • Because of the variation in the use cases of RxFill in LTC, careful consideration should be taken to consume the information provided as sent by the pharmacy.

RxFill Message Flow

The following steps depict the RxFill message flow:

  1. A prescription at the pharmacy is dispensed to a patient or, for LTC, is delivered.
  2. The pharmacy submits an RxFill message via Surescripts to the original prescriber.
  3. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the pharmacy.
  4. Surescripts successfully transmits the RxFill message to the prescriber.
  5. The prescriber's system validates the request and returns a Status, Error (if errors are found) or Verify message.
    1. If a Status 000 is sent in step 5, the prescriber's system would send a Verify.
  6. Surescripts transmits the Status, Error or Verify message back to the pharmacy system.

RxFill Best Practices

RxFill General

  • For prescriptions received by the pharmacy in non-electronic form, the subsequent RxFill messages will not be able to reference an original electronic Message ID or other unique electronic identifier for the prescription. The pharmacy should populate the RxFill message with adequate patient and drug information and the prescriber should be able to match these messages to the appropriate patient and prescription.
  • Prescriber should anticipate and successfully process RxFill messages where the status (dispensed, partially dispensed, not dispensed, or transferred) changes over time. For example, the physician may receive an RxFill (not dispensed) message for a specific prescription today and then receive an RxFill (dispensed) message for the same prescription the next day in the case where a patient forgot to pick up their medication in time but came to the pharmacy again the next day. The RxFill messages should be viewed as a snapshot in time, not a permanent status indicator.
  • Prescriber should anticipate that RxFill messages will only be received for a portion of the prescriptions the physician writes. If no RxFill message is received for a prescription, the system cannot assume that the prescription has not been dispensed or that the patient is not compliant.
    • Prescriber should anticipate receiving an RxFill notification only for prescriptions from pharmacies that support the RxFill transaction.
    • Prescriber should anticipate there may be significant time delays between writing the prescription and the receipt of an RxFill indicating the current status.

RxFill Dispensed

The RxFill message is originated by the pharmacy. This message is used to notify the prescriber when a prescription has been dispensed. This message can be used to notify dispensed status of new prescriptions, refills or medication resupply requests.

  • RxFill (dispensed) message would be sent for all dispensed prescriptions including each refill/resupply, but excluding partial fills, which are represented by a separate message.
  • RxFill (dispensed) message should be triggered by a specific action in the pharmacy that indicates that the prescription was dispensed and not simply filled and waiting. Therefore, the RxFill (dispensed) message should not be triggered simply by label printing or adjudication in the pharmacy, but through an action such as a Point of Sale message, Electronic Signature Capture at pickup, a Will Call Bin Management action, or similar positive indication of actual dispensing. For LTC, RxFill (dispensed) should be triggered by a specific action in the pharmacy that indicates that the prescription was dispensed and/or filled and approved for administration to the patient by the LTC organization.

RxFill Not dispensed

The RxFill message (not dispensed) is originated by the pharmacy. This message is used when the pharmacy has not dispensed a previously filled prescription and the pharmacy will return the medication to stock. This might occur if the patient never picked up the prescription. This message is used in LTC when the pharmacy will not dispense a prescription to notify the LTC organization of the status and reason the prescription will not be dispensed. This message can be used to notify dispensing status of new prescriptions, refills or resupply requests.

  • RxFill (not dispensed) message would be sent when a pharmacy does not successfully dispense or does not intend to dispense a prescription that has been received and processed by the pharmacy, and/or has been waiting for patient pickup.
  • RxFill (not dispensed) message should be triggered by the Return to Stock process of the pharmacy, which indicates the patient has not picked up the prescription within the time period set by the pharmacy for their Return To Stock procedures.
  • RxFill (not dispensed) message should be sent for any prescription (first fill, refill/resupply, partial fill, etc.) that is processed by the pharmacy during Return to Stock procedures.
  • RxFill (not dispensed) message should not be sent by the pharmacy in the case where a prescription is being reversed, but with the intent of subsequent reprocessing and dispensing. An example would be a change in payment coverage requested by the patient at the time of pickup.
  • RxFill (not dispensed) message should not be sent by the pharmacy to indicate that the patient has not requested his/her next refill based on elapsed time since the patient’s previous refill. For example, if the pharmacy dispensed a 30-day supply to the patient but more than 30 days have elapsed and the patient has not requested another refill, the pharmacy system should not automatically generate an RxFill (not dispensed) message. The pharmacy system should use the specific actions of the patient when sending RxFill messages and not inferences in every case possible.
  • RxFill (not dispensed) message in LTC should be triggered by events as determined by the pharmacy at the time of processing. Examples include Drug Use Evaluation conflicts that could not be resolved, the medication order has expired, the pharmacy has no intent to stock the prescribed medication, or a prior authorization was required for the prescribed medication and the payer has denied.
  • Since Return to Stock procedures and timing differ between pharmacies, the timing of the RxFill (not dispensed) message will also vary.

RxFill Partially Dispensed

The RxFill (Partially Dispensed) message is originated by the pharmacy. This message is used to notify the prescriber or LTC organization when a prescription has only been partially dispensed. This message can be used to notify dispensing status of new prescriptions, refills, or medication resupply requests. An RxFill message (partially dispensed) can be repeatedly initiated until the prescription has been completely dispensed or the remaining quantity is returned to stock, thereby requiring an RxFill (not dispensed) message for the remaining quantity. The message requires that, at a minimum, the dispensed drug with the quantity received be included in the message. It is suggested that the prescribed drug be included for clarification.

  • RxFill (partially dispensed) message would be sent for all dispensed prescriptions where only a portion of the specified quantity is provided. Subsequent dispensing of the remainder of the partial fill, or additional partial fills, would also require an appropriate RxFill (partially dispensed) message to be sent.
  • RxFill (partially dispensed) message should be triggered by a specific action in the pharmacy that indicates that the prescription was dispensed and not simply filled and waiting. Therefore, the RxFill (partially dispensed) message should not be triggered simply by label printing or adjudication in the pharmacy, but through an action such as a Point of Sale message, Electronic Signature Capture at pickup, a Will Call Bin Management action, or similar positive indication of actual dispensing.

Census

In LTC, three changes to the patient status occur regularly that drive processes at the pharmacy but are not necessarily tied to a prescription. Because of this, a message type is necessary to convey patient information “decoupled” from any prescription information. The three key use cases that are relevant in LTC are New Admission, Information Change, and Discharge.

The Census message is originated by the facility in an LTC environment. This message notifies the pharmacy about census events and can be used in three cases - to notify the pharmacy of a new resident, a change to demographic information of a resident, or the discharge of a resident.

The differentiation of the three cases is with the use of the 010 Message Function Coded in the REQ Segment.

Census Message flow

The following steps depict the Census message flow:

  1. A resident record is added or updated within a facility system.
  2. Facility system submits a Census message via Surescripts to the pharmacy.
  3. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  4. Surescripts successfully transmits the Census message to the pharmacy.
  5. Pharmacy system validates the message and returns a Status, Error (if errors are found), or Verify message.
    1. If a Status 000 is sent in step 5, the pharmacy system would send a Verify.
  6. Surescripts transmits the Status or Error message back to the facility system.

Resupply

In the LTC environment there is a need to send a Resupply message from a facility to a pharmacy. An example use case is when a medication supply for a resident is running low (2-3 doses) and a new supply is needed from the pharmacy, the nurse needs a way to notify the pharmacy that more medication is needed. Since medication has already been authorized on the original order, the Resupply is simply a request for more inventory, therefore this is not a fillable message like the Refill Renewal Response.

Resupply Message Flow

When it is determined that medication supply is running low, the following steps depict the Prescription Resupply message flow

  1. The Resupply message is successfully transmitted via the facility system to Surescripts.
  2. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  3. Pharmacy system validates the Resupply and returns a Status, Error (if errors are found), or Verify message.
    1. If a Status 000 is sent in step 3, the pharmacy system would send a Verify.
  4. Surescripts transmits the Status, Error, or Verify message back to the facility system.

Cancel Rx

The Cancel Rx Request message is used to notify the pharmacy that a previously sent prescription should be cancelled and not filled. The message is originated by the prescriber system as a Cancel Rx Request message.

The Cancel Rx Response message is sent from the pharmacy to the prescriber system in response to a Cancel Rx Request message.

Cancel Workflow One

  1. Patient has not picked up the medication; pharmacist cancels the prescription from the system and returns it back to stock. Pharmacist also sends back a Cancel Rx Response message to the prescriber.
  2. In this scenario, pharmacy should send an "A" (Approval) code back in the Cancel Rx Response message to the prescriber. Sending this code back would clearly indicate to the prescriber that the pharmacy has cancelled the entire prescription and no medication was dispensed to the patient.
  3. Prescriber systems should maintain a record within their system, i.e. patient prescription profile to document that a Cancel Rx Request was initiated along with the acknowledgment and the action taken by the pharmacy.
  4. Pharmacy customers should update the patient medication profile within their systems to document along with the appropriate reasons.

This would clearly indicate the number of prescription refills that were dispensed and that there are no remaining refills left. Pharmacy should mark the prescription cancelled so that a refill request will not be sent at a later date.
  3. Prescriber systems should maintain a record within their system, i.e. patient prescription profile to document that a Cancel Rx Request was initiated along with the acknowledgment and the action taken by the pharmacy.

  1. Pharmacy can't identify the patient or the patient in the pharmacy system doesn't have the medication that the prescriber wants to cancel.
  2. In this scenario, pharmacies should send a "D" (Denied) code back in the Cancel Rx Response message to the prescriber. Sending this code back would clearly indicate to the prescriber that the pharmacy hasn't cancelled the prescription. Pharmacies should also include the denial reason or denial reason code to further explain the reason.
  3. Prescriber systems should maintain a record within their system i.e. patient prescription profile to document that a CancelRx Request was initiated along with the acknowledgment and the action taken by the pharmacy.

This would clearly indicate that partial fills were dispensed to the patient but all remaining refills have been cancelled.
  3. Prescriber systems should maintain a record within their system, i.e. patient prescription profile to document that a Cancel Rx Request was initiated along with the acknowledgment and the action taken by the pharmacy.
  4. Pharmacy should update the patient medication profile within their systems to document along with the appropriate response.

Scenario 2:

  1. Patient has picked up all the medications. There is nothing left in the queue to be canceled. Pharmacist sends back a Cancel Rx Response message to the physician.
  2. Pharmacy should send an "A" (Approval) code in the Cancel Rx Response message with a Note that specifies how many refills have been dispensed already. This would clearly indicate the number of prescription refills that were dispensed and that there are no remaining refills left. Pharmacy should mark the prescription cancelled so that a refill request will not be sent at a later date.
  3. Prescriber systems should maintain a record within their system, i.e. patient prescription profile to document that a Cancel Rx Request was initiated along with the acknowledgment and the action taken by the pharmacy.

Scenario 3:

  1. Pharmacy can't identify the patient or the patient in the pharmacy system doesn't have the medication that the prescriber wants to cancel.
  2. In this scenario, pharmacies should send a "D" (Denied) code back in the Cancel Rx Response message to the prescriber. Sending this code back would clearly indicate to the prescriber that the pharmacy hasn't cancelled the prescription. Pharmacies should also include the denial reason or denial reason code to further explain the reason.
  3. Prescriber systems should maintain a record within their system, i.e. patient prescription profile to document that a Cancel Rx Request was initiated along with the acknowledgment and the action taken by the pharmacy.

CancelRx Prescription Message Flow

The following steps depict the Cancel Rx message flow:

  1. Prescriber initiates a Cancel Rx Request for a patient via a prescriber system.
  2. Prescriber system successfully transmits the Cancel Rx Request to Surescripts.
  3. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the requester.
  4. Surescripts utilizes the Recipient ID and routes the message appropriately.
  5. Pharmacy system validates the Cancel Rx Request and responds with a Status, Error (if errors are found), or Verify message.
    1. If a Status 000 was sent in step 5, the pharmacy system would send a Verify.
  6. Surescripts transmits the Status, Error, or Verify back to the sender.
  7. Pharmacy processes the Cancel Rx Request and returns the prescribers' Cancel Prescription Response.
  8. Surescripts validates the message. If there is an issue with the message, Surescripts sends an Error message back to the pharmacy system.
  9. Surescripts transmits the Cancel Rx Response message back to the prescriber system.
  10. Prescriber system validates the Cancel Rx Request Response and returns a Status, Error (if errors are found), or Verify message.
    1. If a Status 000 is sent in step 10, the prescriber system would send a Verify.
  11. Surescripts transmits the Status, Error, or Verify back to the pharmacy system.

CancelRx General Best Practices

  1. Cancel Rx Request can be sent for paper or electronic.
  2. Cancel Rx Request should not be faxed.
  3. Cancel Rx Response time – If no response after 12 hours the prescriber should call the pharmacy.
  4. Pharmacy should always treat incoming Cancel Rx Request messages as top priority and as such display them prominently to their end users. If the Cancel Rx Request message has already been processed, further action may be required, but it doesn't need to be treated as top priority.
  5. Prescribers should not send a Cancel Rx Request for a prescription that is expired based on federal or state regulations. There should be programmatic checks in place to allow a Cancel Rx Request up to the expiration date of the prescription based off of the written date of the prescription. For example, the DEA requires Controlled Substance Rx to be filled within 6 months from the date written, and most states limit the filling of non-controlled Rx's to 1 year from the date written.
  6. Pharmacy should always include a Note in an "A" (Approval) Cancel Rx Response message when responding to a Cancel Rx Request message if the patient has picked up the "first fill" or any refills thereafter and the pharmacy is cancelling the remaining refills left on the prescription.
  7. Pharmacy should not send any Refill Renewal Request message after a Cancel Rx Request message is received.
  8. A prescriber who has assumed responsibility for the patient's care may potentially cancel any prescription, even if they are not the original prescriber. It remains up to the pharmacy to determine if the Cancel Rx Request from the prescriber is appropriate.
  9. The Cancel Rx Request should contain the RxNorm in the <DrugCoded>.
  10. The Cancel Rx Request should contain pertinent information for the pharmacy to be able to find the prescription in their system (patient, medication (name, strength, and dosage form), and prescriber). If the pharmacy cannot definitively determine the prescription to be canceled, manual processes will occur to verify the cancellation.
  11. Pharmacies should not delete a Cancel Rx Request message from a processing queue without a response being generated to the requestor. Pharmacy edits should be put in place to not allow a medication to be provided to the patient if a Cancel Rx Response has not been sent.
  12. The prescriber should notify the patient or caregiver to inform them of the cancellation of a prescription. The Cancel Rx Request was not intended to relieve the prescriber of the responsibility of notifying the patient or caregiver to advise of the drug therapy change – it is only intended as a backup to prevent inadvertent drug therapy continuation or resumption at a later date.
  13. Pharmacy should provide clear denial reasons in Cancel Rx Response denial responses. Note the Description of <DenialReasonCode> is the description of the value defined in the NCPDP External Code List. If the <DenialReasonCode> is sent, the <DenialReason> should not contain the echoing of this description as it adds no information.

Verify

This NCPDP SCRIPT message is returned only in an asynchronous response. The Verify message confirms delivery of a message to its final destination. The Verify message is an NCPDP message that indicates the acceptance of the request. This message is used to communicate the data content status of a message and to indicate the delivery of the message to its final destination.

A Verify indicates the process is complete. Subsequent acknowledgment messages should not be expected.

Verify (XML)

The VerifyStatus shown below is a Complex Type made up of code and description.

Error

This message indicates that an error has occurred. An error can be generated when there is a communication problem or when an error occurs during the processing of the message.

An Error message can either be free-standing (asynchronous) or as an HTTP response.

An Error indicates the process is complete. Subsequent acknowledgment messages should not be expected.

A synchronous error is sent in response to a message before breaking the communication connection. An asynchronous error message (also called Free Standing Error message) is sent in response to a message any time after the Status 000 message is sent.

An Error message may not be used to respond to an Error message.

Please refer to the 900 Error Text Table for EPCS.

Status

The Status message is used to indicate the message was accepted of a message back to the sender. A Status message is used in response to a NewRx, Refill Renewal Request, Rx Change Request, Rx Change Response, Cancel Rx Request, Cancel Rx Response, RxFill, Resupply, Census or Refill Renewal Response to indicate acceptance and responsibility for a request.

The Status message is an NCPDP message that indicates the acceptance of the request. Surescripts is utilizing the Status message for all messages.

This message is used to indicate the message was accepted for processing.

A Status message can only be used in the HTTP response to a message synchronously.

A Status code of 010 indicates the process is complete. Subsequent acknowledgement messages should not be expected.
