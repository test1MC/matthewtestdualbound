Messages Overview

Message Format

Customers must create and send syntactically valid messages and shall be able to receive syntactically valid maximum and minimum populated messages. Optional data elements (and values therein) will not cause message failure.

For full message descriptions, see the NCPDP Implementation Guide.

Surescripts Prescription Routing service enables new prescription transmissions from a prescriber to the patient's pharmacy of choice. It also includes the ability to facilitate renewal requests/responses and prescription change requests/responses, etc. between pharmacies and prescribers. Detailed Prescription Routing Message Workflow

Scenario 1

1a). A NewRx message is sent to Surescripts.

1b). Surescripts cannot determine what format the message is in and sends an error text back to the sending customer.

Scenario 2:

2a). A NewRx message is sent to Surescripts.

2b). Surescripts recognizes the format as NCPDP but finds errors in the message. Surescripts sends back an Error message.

Scenario 3:

3a). A NewRx message is sent to Surescripts.

3b). Surescripts forwards the message on to the receiving customer.

3c). Receiving customer cannot recognize the message and sends an Error text to Surescripts.

3d). Surescripts creates an Error and sends it back to the sending customer.

Scenario 4:

4a). A NewRx message is sent to Surescripts.

4b). Surescripts forwards the message on to the receiving customer.

4c). Receiving customer validates the message and sends back a Status (code 010) message to Surescripts. Code 010 means the message was accepted by ultimate receiver.

4d). Surescripts forwards the Status (Code 010) message back to the sending customer.

Scenario 5:

5a). A NewRx message is sent to Surescripts.

5b). Surescripts forwards the message on to the receiving customer.

5c). Receiving customer validates the message, finds errors in the message, and sends back an Error message to Surescripts.

5d). Surescripts forwards the Error message back to the sending customer.

Scenario 6

6a). A NewRx message is sent to Surescripts.

6b). Surescripts forwards the message on to the receiving customer.

6c). Receiving customer validates the message and sends back a pending Status (Code 000) message to Surescripts.

6d). Surescripts forwards the pending Status (Code 000) message back to the sending customer.

6e). Receiving customer fails trying to process message and initiates a new Error message (sometimes referred to as a "Free Standing Error") to Surescripts.

6f). Surescripts forwards the Error to the sending customer.

6g). Sending customer acknowledges the Error by sending back a verified Status (Code 010) message to Surescripts.

6h). Surescripts forwards the verified Status (code 010) to the receiving customer.

Scenario 7

7a). A NewRx message is sent to Surescripts.

7b). Surescripts forwards the message on to the receiving customer.

7c). Receiving customer validates the message, and sends back a Status (Code 000) message to Surescripts. In this case the message has not reached its ultimate destination.

7d). Surescripts forwards the Status (Code 000) message back to the sending customer.

7e). Receiving customer delivers the NewRx to the end point and sends back a Verify.

7f). Surescripts forwards the Verify to the sending customer.

7g). Sending customer acknowledges the Verify by sending back a Status (Code 010) message to Surescripts. Code 010 means it was accepted by the ultimate receiver.

7h). Surescripts forwards the Status (Code 010) to the receiving customer.

Where the Status message is used, the status type code can be either 000 – Transmission successful or 010 – Successfully accepted by ultimate receiver. The 000 indicates that there could be an Error or Verify forthcoming. If an Error or Verify is not received following a Status 000, then no assumptions should be made on whether the message has been received successfully or not. The 010 indicates that no further messages will follow

EPCS Message Workflow

All e-prescribing applications sending Electronic Prescriptions for Controlled Substances are required to provide proof of successfully completing a third party audit according to the DEA requirements. This proof of successful audit is required before the application is able to send messages across the Surescripts network.

The EPCS flow for fillable messages (NewRx, Approved/Approved with changes Refill Response, and Approved/Approved with changes Change Response) follows the flow above with these additions:

  • The service level for both the prescriber and pharmacy must be enabled for controlled substances. Otherwise the message will be rejected.
  • The message must be digitally signed and contain either the signature or the flag indicating it has been signed. Surescripts will not validate the signature.
  • The message must contain at least one of the following: NDC, Compound Flag, or Supplies Flag.

Message Validation

State Specific Validation for Controlled Substances

Surescripts will stop controlled substance messages from being sent or received that are prohibited by state law as outlined below. Technology vendor systems and their users are still ultimately responsible for continued compliance with all regulations.

Each message without a DEA Schedule will have the NDC checked to determine if the medication is a controlled substance at a federal or state level. Once a medication has been identified as a controlled substance, either federally or by the state it is written in or being sent to, the following validation checks occur:

  • Is the sender/receiver authorized to send/receive EPCS messages?
  • Is the Signature Indicator (SI) flag or Digital Signature on the message?
  • Is the DEA Schedule field populated?
  • If NDC is not present, is there a compound or supply flag?
  • Does the state the prescription is written in/sent to allow e-Prescribing of Controlled Substances?
    • If the medication is classified as a schedule II controlled substance, does the state allow e-Prescribing of schedule II controlled substances?
    • For mail order pharmacies Surescripts will not check the NDC against the location of the mail order pharmacy. Mail order pharmacies are still required to perform the appropriate state validations.

Failure Mode/Response Approach

Surescripts has identified different levels of failure for NCPDP-based messages:

  • In instances where Surescripts or a customer receives a message that is unrecognizable, the recipient should send back an error text.
  • If an error occurs after the sender has been identified and validated, and the message has been determined to be an NCPDP message, an NCPDP Error message is returned.
  • If the recipient identifies an error, an Error message is created and passed through Surescripts.

Faxing

Faxable Error Messages (FEM) service delivers non-controlled substances via facsimile (FAX) when they cannot be delivered electronically due to temporary/transient transmission failures, unless prohibited by state law. Surescripts creates an acknowledgment (Status-000) and delivers it back to the sending customer when putting the message in a fax queue. The sending customer will receive verification (Verify-010) when the fax is successfully delivered or an Error if fax delivery fails.

In previous versions Surescripts faxed controlled substance prescriptions back to the prescriber for a wet signature. For EPCS, this feature will no longer be supported. If a prescription for a controlled substance cannot be delivered electronically, the sending customer will receive an Error.

Currently Surescripts only supports faxing for NewRx, Refill Renewal Request and Refill Renewal Response within Prescription Routing.
