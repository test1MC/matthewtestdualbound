Implementation of EPCS

Application vendors and pharmacies will be allowed to transact Electronic Prescribing of Controlled Substances (EPCS) on the Surescripts network when they have completed the following:

1. Updated e-prescribing software to meet all requirements specified by the DEA in the Interim Final Rule and the SCRIPT messaging version (this guide) that support EPCS.

2. Undergone a third-party audit to ensure the software meets all DEA EPCS requirements.

3. Achieved Surescripts Certification to the Surescripts Implementation Guide.

4. Make audit results available to Surescripts with a Proof of Audit Letter & Surescripts EPCS Audit Attestation Form.

Prescription Routing General Best Practices

The following general best practices apply to all prescription routing message types.

  1. The application should make clear to the user, which data elements will be sent to the receiver and which elements are for internal/local use.
  2. If customers implement any type of prescription message store and forward methodology, e.g. hub or central location, the storage/hold time period should be no more than 1-2 minutes.
  3. To maximize the use of electronic transmission, customers should default to electronically send within their application workflow when the receiver is electronically enabled.
  4. Application should not duplicate messages; i.e. electronically delivered as well as faxed.
  5. Applications should have the ability to send ICD-10 information in all message types. This is because ICD-10 coding is required on some electronic claims. If the information is not included, it could result in a manual process between the provider and pharmacy.
  6. Notes and other free-text fields should not contain information for which there is a designated field.

XML Escape Characters

Most symbols should be ‘safe’ within most alphanumeric fields; however, XML validation has issues with some characters, such as ‘less than’ (<) and ‘ampersand’ (&); therefore, these symbols should be properly escaped or un-escaped. The quote and apostrophe only need to be escaped when used in an attribute. The most common escape encodings are:

ASCII Characters

XML Escape

quote (")

&quot;

apostrophe ('‘)

&apos;

ampersand (&)

&amp;

less than (<)

&lt;

greater than (>)

&gt;

String lengths are calculated based on the un-escaped value. For example, “>98 degrees” should be encoded as “&gt;98 degrees” but it would only be counted as a string length of 11, not 14. The quote and apostrophe only need to be escaped if they are used within an attribute.

Timeouts

For timeouts, consider the following:

  • When sending a message to Surescripts, the initiator should set the http timeout to no less than 30 seconds.

  • A receiving system must reply with a valid Status/Error response within 24 seconds.

  • When a Status 000 is sent as a reply, a receiving system must follow-up with a Verify or Error message within 24 hours, as Surescripts may return a timeout error.

  • To avoid duplicate requests, a sending system should wait at least 48 hours before retrying RxRenewalRequest if no RxRenewalResponse has been received.

General Display Requirements

Surescripts understands our customers are in the best position to make workflow design decisions and we are committed to encouraging innovative designs. Surescripts is also committed to ensuring the highest level of patient safety possible. If a workflow could potentially result in a patient safety issue, or if a patient safety issue is identified, Surescripts may require application code changes to ensure positive patient outcomes.

To ensure patient safety, the application shall take steps to ensure that the following critical fields (as will be transmitted in the message) are reviewed by the sender for accuracy and displayed to the receiver.

The code value for codified fields does not have to be displayed, only the description.

  • Patient First Name
  • Patient Last Name
  • Gender
  • DOB
  • Drug Description
  • Quantity/PUC
  • SIG (free text sig vs. structured)
  • Notes (to/from pharmacy)
  • Refills
  • Days Supply
  • Pharmacy Name
  • Effective Date (if in future)
  • Height and Weight, if applicable

Any element that is not sent with the transaction shall R6.041 be labelled “For internal use only” on the summary screen, so the prescriber knows that information will not be sent with the message.

When a field is not sent, it shall R6.155 not be defaulted to display 0.

To allow for patient choice, the application shall allow a pharmacy search based on pharmacy type of mail/retail at a minimum; other pharmacy search options may include specialty and long term care.

The application shall not filter out pharmacies of a specific type when a search by geography (city, state, and zip code) is performed, unless a specific pharmacy type is requested. The application shall display the following pharmacy information in the pharmacy search results: Pharmacy Name, Street address, City, State, Zip code, phone number and pharmacy type.
