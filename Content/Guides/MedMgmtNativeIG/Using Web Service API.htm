Using Web Service API

The Web Service API allows access to functionality via RESTful web services for customers who prefer to develop their own user interface.

General Conventions

The Web Service API follows Representational State Transfer (REST) architecture principles using HTTP requests with appropriate HTTP verbs and status codes.

The message body for requests and responses are of type application/json or application/xml.

Versioning and URLs

URLs are of the form https://{{host}}/uiservice/api/{{version}}/{{path}} or , https://{{host}}/mmpoc/api/{{version}}/{{path}} where:

  • the host depends on the environment and is provided  as part of the Surescripts integration process;
  • version is the requested version of the endpoint and is provided as part of the Surescripts integration process
  • path is the path to the specific service

For example: https://{{host}}/uiservice/api/v0/worklist

The API version number ("v0" in the above example) is incremented when a breaking change is introduced. A breaking change prevents an existing implementation from communicating with a particular version. Adding additional fields is not considered a breaking change. An API client application needs to support additional fields being added to an existing version.

Examples of breaking changes includes, but are not limited to:

  • Changing type or attributes of an existing field
  • Changing an endpoint path
  • Changing the semantic meaning of an existing field

Summary of API Endpoints

The following provides a summary of API endpoints and the data that they provide.

API Endpoint

Description

GET worklist

Retrieves a paged list of medication management messages for a given patient.

POST  worklist

Retrieves a paged list of the active tasks for a given collection of recipients (providers).

GET worklist/summary

Retrieves the count of worklist tasks for the patient.

POST  worklist/summary

Retrieves the count of worklist tasks for the provider.

GET   adherence-profiles/summary

Retrieves the patient’s lowest PDC ratio.

GET  patient-ids

Searches the Surescripts PBM/payer MPI and retrieves the pbmMemberId and payerId. Note: This is used instead of the eligibility 271 in workflows that are not associated with a patient visit/encounter.

GET CommunicationRequest

Retrieves the medication management message details for a task_id

GET  Observation/

Retrieves the patients adherence profile ( PDC ratios, medications, last fill date)

POST QuestionnaireResponse/

Submits the response from the EHR end user to the medication management message

User Security

Web Service API requests are sent using HTTPS and routed through a whitelisted vendor aggregation server.

A call to a Web Service API requires the following headers:

Header Field Name

Description

Example

Authorization

A standard basic authentication header using the credentials assigned during the integration process. 

Example:Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==

X-SENDER-UID

A custom header that contains a Surescripts identifier assigned during the integrations process. The format for the header value is <qualifier><id>. 

Qualifier: DPExample:X-SENDER-UID: DP-ABC1

Content-Type

The MIME type of the body of the request  

Values: application/xml, application/json+fhirExamples: Content-Type: application/xmlContent-Type: application/json+fhir

Accept

The MIME type of the body of the response  

Values: application/xml, application/json+fhirExamples: Accept: application/xmlAccept: application/json+fhir

x-SPI

The SPI of the provider end user or the provider that the end user is working on behalf of.  Multiple values are separated by a comma.Used with GET CommunicationRequest and Observation APIs

Examples:x-SPI: 6598607334001x-SPI: 6598607334001,6390518189001

x-user-role

The end user role. Multiple values are separated by a commaUsed with GET CommunicationRequest and Observation APIs

x-user-role: mmpocsubmitter,mmpocviewer
