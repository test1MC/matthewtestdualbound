Application Certification Requirements

Formulary and Benefits

General Requirements

RTF Implementation F.100: Customers implementing RTF will still need to implement an industry standard drug database listed above. It is not necessary to change databases as long as one of the above is presently implemented. RTF customers will send a Drug Database identifier on Benefit Request messages so that Surescripts can return the appropriate information on Benefit Response messages.

RTF Implementation F.101: RTF will return Brand/Generic, OTC/Rx indicators for the requested medication and Alternatives. Previously, this information was used to determine the unlisted formulary status for Payer provided alternatives. Surescripts will now make this determination via RTF.

Presentation of Formulary Status

RTF Implementation F.105: RTF will return formulary information at the Representative NDC (dispensable medication) level. If the provider vendor chooses to display formulary status at a higher level, requirement F.105 shall be fulfilled.

RTF Implementation F.106: An RTF Benefit Request shall not be submitted when the 271 Eligibility Response does not contain active coverage information.

F.107: The application shall display a formulary status of "Unknown" for all medications when there is no formulary status list ID or corresponding formulary status list for the coverage selected.

RTF Implementation F.107: An RTF Benefit Request shall not be submitted if the 271 Eligibility Response does not contain a Formulary ID. In this instance, if the Benefit Request is sent, Surescripts returns an NCPDP ERROR (900/3000/Invalid Syntax. Message fails schema validation.) If the Benefit Request contains a Formulary ID that is not found at Surescripts, the Benefit Response returns a Reason Code of "NotProcessed" and "Unknown Formulary ID" in the Free Text field.

Presentation of Formulary Alternatives Coverage and Copay

F.108: For active coverages, during the prescription writing process and prior to the summary screen, the application shall display, at a minimum, alternative medications when the target medication is an On-Formulary/Non-Preferred, Non-Formulary, or Non-Reimbursable medication.

RTF Implementation F.108: When the requested medication is 2 - On-Formulary/Non-Preferred, Non-Formulary, or Non-Reimbursable (single-NDC Request), RTF will return payer provided alternatives or therapeutic alternatives (as identified by the provider vendor) of a higher formulary status.

A provider vendor may display Therapeutic alternatives if a Payer Alternative List is present, but an alternative match for the target NDC does not exist.

Best Practice

  • The application may further filter alternatives by utilizing decision support logic (e.g. only showing alternatives applicable due to gender, age, diagnosis, etc.).
  • If the target medication is a brand medication and the prescriber indicates that generic substitution is permitted, the application does not need to show alternatives if the generic equivalent has a higher formulary status.

Payer Provided Alternatives

  1. If the payer provides a Formulary Alternatives List, the application shall show all alternatives that have a higher On-Formulary status.

    1. For Non-Formulary or Non-Reimbursable medications, show alternatives that are On-Formulary/Non-Preferred or On-Formulary/Preferred.

    2. For On-Formulary/Non-Preferred medications, show On-Formulary/Preferred.

    3. Do not show alternatives for Unknown medications.

  2. Show all strengths available that have a higher formulary status for the NDC provided in the Formulary Alternatives List.

RTF Implementation: When a payer provided alternative is returned in an RTF Benefit Response, Surescripts will return all strengths of that alternative with a higher formulary status. All alternatives in the Benefit Response shall be displayed.

  1. The alternatives provided in a Formulary Alternatives List shall be ordered according to the stated preference level in that list from high to low.

  2. Show all alternatives in the Formulary Alternatives List, even if the target medication is an OTC, multi-ingredient medication, or supply.

  3. The application shall not show drug database-provided alternatives if the PBM provides a Formulary Alternatives List.

RTF Implementation: Customers shall send appropriate therapeutic alternatives on every single-NDC Benefit Request. If payer provided alternatives are present, Surescripts will remove the therapeutic alternatives and replace them with the payer provided alternatives on the Benefit Response. If payer alternatives are not provided, Surescripts will return necessary formulary information for the therapeutic alternatives sent in the Benefit Request.

  1. The application shall display formulary status and full or abbreviated retail copay data for all alternatives. Retail copay (defined as Pharmacy Type Retail, Any, or Null) data displayed shall be, at a minimum, the Copay Tier, Flat Copay Amount, or Percent Copay Rate.

If the application cannot display the complete list of alternatives on one page, the application may allow the user to take action to view additional alternatives.

Drug Database Provided (Therapeutic Alternatives)

  1. If the PBM does not provide a Formulary Alternatives List, the application shall show all commercially available, generic, and branded alternatives with a higher On-Formulary status in the same therapeutic class as the target medication as determined by an approved medication data source.

    1. For Non-Formulary or Non-Reimbursable medications, show alternatives that are On-Formulary/Non-Preferred or On-Formulary/Preferred.

    2. For On-Formulary/Non-Preferred medications, show On-Formulary/Preferred.

    3. Do not show alternatives for Unknown medications.

RTF Implementation: Customers shall send all appropriate therapeutic alternatives on all single-NDC Benefit Request messages. Surescripts will respond with formulary information for any alternatives of a higher formulary status.

Best Practice

  • If the target medication is OTC, the application should not display alternatives.

  • If the target medication is a Supply, the application should not display alternatives.

  • If possible, the application should only show alternatives for the same route of administration as the target medication.

  1. If the target medication is single-ingredient, the application shall only display single-ingredient alternatives.

  2. If the target medication is multi-ingredient, the application shall only display multi-ingredient alternatives.

  1. The application shall display formulary status and full or abbreviated retail copay data for all alternatives. Retail copay (defined as Pharmacy Type Retail, Any, or Null) data displayed shall be, at a minimum, the Copay Tier, Flat Copay Amount, or Percent Copay Rate.

If the application cannot display the complete list of alternatives on one page, the application may allow the user to take action to view additional alternatives.

RTPB Workflow

Benefit Request

PM.100: Benefit Requests are to be sent only to active PBM customers whose directory service level supports RTPB.

PM.101: During the prescription writing process and prior to sending the NewRx when a Benefit Request is run, the Benefit Request shall include one target pharmacy and one NDC for one active patient coverage.

Benefit Response

Display Requirements

PM.105: During the prescription writing process and prior to sending the New Rx, the application shall display:

PM.106: The Estimated Patient Pay Amount, Quantity Priced, and Days Supply Priced at requested pharmacy for all medications returned (including alternatives).

PM.107: Pricing at other Pharmacy Types. This concession is allowed: An indicator that pricing at other Pharmacy Types apply. User action on this indicator presents all pricing at other Pharmacy Types in their entirety.

PM.108: Coverage Factors for all Pharmacy Type and medication combinations returned (including alternatives). This concession is allowed: An indicator that Coverage Factor(s) apply to a specific combination. User action on this indicator presents all Coverage Factors in their entirety.

For more information, see the Coverage Alert Reference Codes and Text section.

PM.109: All available appropriate alternatives displayed together. If all alternatives cannot be shown together, this concession is allowed: An indicator that alternative medications are available. User interaction on this indicator presents all alternative medications together.

PM.110: User shall be made aware that the pricing/coverage data is a point in time calculation based upon the Quantity and Days Supply provided in the Request and may vary once the prescription is filled at the dispensing pharmacy. Any returned alternative information is informational only and not intended to replace clinical decisions.

PM.111: Alternatives shall be displayed in the order provided from the PBM. First alternative in response will be first alternative displayed.

PM.112: Pricing information shall be displayed as returned in the Benefit Response. Advertisements, coupons and other discount program information shall not be used to influence or attempt to influence, through economic incentives or otherwise, the Prescribing Decision of a prescriber at the Point of Care.

General Requirements

PM.115: Provider vendor shall be certified on Surescripts' Directory, Eligibility, Formulary, and Prescription Routing to implement the Benefit Request and Response (RTPB workflow).

PM.117: Customers shall support the scenarios in the Message ID Linkage section of this Guide.
