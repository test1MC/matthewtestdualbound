Error Message Processing Summary

This section explains the message processing that takes place for messages exchanged between a message sender and a message receiver. These messages are initiated in the form of a synchronous request and a response messages. The sender stays connected until the receiver sends a response. Depending on connectivity, the actual message vehicle may vary.

The system (Surescripts) stores round-trip messages until the receiver responds to the message or until the specified time limit has elapsed. If the timeout elapses before the message is processed, Surescripts will return an error message to the sender as the reply (explained below). If there is a desire to re-try a Benefit RequestPMBC Request, the provider will need to send a new message. Surescripts does not run a re-try in the background.

Note: The table below represents errors that are due to syntax or connectivity issues. There are other errors that may be returned in the response in one or more of the following elements:

  • Response/ReasonCode
  • Response/FreeText
  • PatientSpecificPricingCoverages/PricingCoverage/DrugStatus/DrugStatusCode
  • PatientSpecificPricingCoverages/PricingCoverage/DrugStatus/FreeText

The Reason Codes and Drug Status Codes are in the ECL schema.

Errors are not necessarily processed in the order as shown below. The row number is there to help the reader indicate which error code they may be discussing.
