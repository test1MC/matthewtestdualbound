Prior Authorizations Managed by a Third Party Processor

Some PBM/payers use a third party processor to manage all, or a portion, of their prior authorization processing. If this is the case, the PBM/payer will return a response with a reason code indicating that they do not process PAs for the patient/medication combination (reason code CP) or that they do not process PAs for the patient (reason code CO). Surescripts is able to perform a search for the third party processor if one of these two reason codes are returned. If a processor is found, Surescripts will send a PAInitiationRequest on behalf of the provider vendor for the user. The response that is received from the third party processor will be returned to the user in the same manner that a response from a PBM/payer would be returned. In the event that the third party processor does not support ePA or Surescripts is unable to find a third party processor, the user will receive a PAInitiationResponse indicating that.

Reviewing Prior Auth Notes from Multiple Entities

PBM/payers, third party processor and Surescripts may include additional free-text details with their ePA responses. Users are able to review the details in order to determine the most appropriate or required action for an ePA by referring to these PA Notes. Each message will have its own designated area for notes.

On subsequent responses for the same prior authorization (if the original PBM/payer was not the processor or does not support ePA), users can go to the process details from the Task History or use the Task History link from within the task details screen. Selecting the View option next to PA Notes will bring up a chronological view of all PA Notes received in response to the selected ePA.
