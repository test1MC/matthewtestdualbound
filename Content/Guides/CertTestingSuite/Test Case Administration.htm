Test Case Administration

The Test Case Administration screen allows you to search for test cases, create new test cases, and edit /delete existing test cases. Two types of test cases are available for use: Certification Base and Portal Related.

Certification Base Test Cases are intended to be a starting point of base test cases that are available for use. Start with these instead of creating a test case from scratch. These can be cloned.

Portal Related Test Cases are those test cases that the user has saved to their portal.

From the Certification Suite>Main window, select the Test Case Administration link on the toolbar or click the Test Case Administration icon from the quick access toolbar as shown below.

The Test Case Administration window opens.

The above screen shows the search controls and results of a successful search. To narrow or change the search results for test cases, specify different values for the search parameters.

The following table lists the message search filters. These search filters are used throughout the suite.

Message Search Filters

Control Title

Description

Test Case Name

This control allows the user to specify a “Begins With” match for the test case name for which they are searching. If this field is left blank all test cases that match the other filters are returned.

Portal

This drop-down list displays the portals associated with the user. Users will also have access to the Certification Base test cases from the Portal drop-down, which can be cloned for use.

Test Case Group

This drop-down filters by test group names that the user has created for test cases relating to their respective portals.

Message Type

This drop-down list filters by Message Type and defaults to Any Message Type if no other type is specified.

If selected, the list returns test case search results that have been saved as “Active” for the associated portal.

To search by test case name, enter part or the entire test case name into the Test Case Name field and click Search. The page refreshes with the filtered results. The Search results contain the following basic fields of information relating to the test case: Test Group, Test Case Name, Portal, Active State, Message Type, Date Created, Date Last Modified, Last Modified By, and Clone.

Edit/Create Test Case

Go to Certification Suite>Test Case Administration>New Test Case. The Edit/Create Test Cases window opens. This window allows you to edit existing test cases, create new test cases, or clone test cases. The Edit/Create Test Case window has three sections as shown in the screen image below. In this image the General Information section of the screen is expanded. Each of these sections is described in further detail in the table below.

  • General Information
  • Editable Raw Test Case Body
  • Other Information

You can manipulate the values in the General Information and Editable Raw Test Case Body sections; the Other Information section provides read-only data for review.

Edit Test Case Sections

Section Title

Description

General Information

The General Information section has several key fields, some to be completed and others are informational about the message body.

Name: Assign a name for the test case

Test Group Name: Assign a Test Group Name if desired.

Portal: Select /assign the test case to a portal.

Partner Type/Message Type: Populated by the system once a message has been successfully loaded into the page.

Active: When selected, the test case is saved to a list of test cases and is available for later use. When not selected, the test case defaults to Inactive and may be recalled for up to 30 days by selecting Inactive from the Message Type drop-down.

Editable Raw Test Case Body

The Editable Raw Test Case Body region provides a test box where the contents of the current test case can be edited. Click Validate upon completing changes to the test case or it will be validated against the schema upon upload or an attempted save of the test case as well.

If the validation is successful, the test case is loaded and all related controls are populated.

If the validation or loading of the xml test case fails, a message notification displays at the top of the page noting what is wrong with the test case.

Other Information

This region displays non-editable information relating to the test case’s last editor, the date created, and the last date/time the case was modified. The information displayed here is controlled by the application.

Creating a New Test Case

To create a new test case, go to Certification Suite>Test Case Administration and select the New Test Case link above the search filters.

The Edit/Create Test Case window opens with a blank form.

Add the information to create a new test case or you may choose to clone or edit an existing test case. For cloning or editing a test case, see below.

Editing an Existing Test Case

To edit an existing test case, go to Certification & Testing Suite>Test Case Administration and click on the <test case name> link for the test case you wish to edit from the Search results section.

The Edit/Create Test Case window opens with the existing test case information populated in the fields, as shown below.

You may choose to replace patient or medication information when editing or cloning a test case. Please refer to Profile Management on page 1 or Medications Management on page 1 for more information.

Cloning a Test Case

As mentioned earlier, most users do not have the ability to edit Certification Base test cases; however, the Certification Base test cases may be cloned to a user’s portal. Cloning permits you to make a copy of an existing test case, modify values found in the test case, and then save the test case to an existing, controlled portal.

Go to Certification & Testing Suite>Test Case Administration. The Test Case Administration window opens. Select the test case you would like to clone, or if no test cases are listed in the search results portion of the screen, select [Certification Base] from the Portal drop-down list. To refine your search further, select the message type from the Message Type drop-down (default is Any Message Type) or sort the list using one of the column heading. A list of Certification Base test cases displays.

Locate the test case you would like to clone and click the Clone link in the far right column for that test case.

The Edit/Create Test Case window opens with the values pre-populated from the cloned test case. Name your cloned test case in the Name field and complete other fields as needed. When finished with the edits to the cloned test case, click Save and it will be saved to your portal. To validate the test case before you save, click the Validate button – the test case will be validated when you save it as well.

You may choose to replace patient or medication information when editing or cloning a test case. Please refer to section Profile Management on page 1 or Medications Management on page 1 for more information.

Test Case Execution

There are two test case wizards; one for Non-Response Based Messages and one for Response Based Messages. Both message types are explained in more detail below.

Non-Response Based messages are any message type that can be considered a request or that does not have any associated response message. These messages include: NEWRX and REFREQ.

Response Based Messages include all messages that derived from an existing request message, such as REFRES and Error messages.

To execute a test case, go to Home>Certification Suite and click on the <test case name> from the Search results list you would like to execute.

You may also search for a particular test case or use the filters to narrow your search results as shown in the screen below.

The execution wizard for the selected message type opens. The steps for building out each type of test message are described below.

Non-Response Based Messages

When this type of message is selected from the search results screen on the Certification Suite Main screen, the execution wizard opens to take you through the process of setting up the message for delivery to the portal you are impersonating. Every step in the wizard contains a brief description of expected results. If an error is encountered during a step, a message will explain what to do next. The wizard steps are shown below. Note that the name of the message you are executing is at the top of each step within the wizard.

Step 1: Introduction

Step 1 describes the wizard. You may want to confirm you are impersonating the correct portal to which you wish to send the message. Click Next Step.

Step 2: Sending Portal Selection

Select the portal from the From Portal drop-down. This is the test portal from which the message is to be sent. These portals are limited and controlled by the application. In most cases the portal will be the opposite type from that which you are impersonating; i.e. if impersonating a prescribing participant, a pharmacy test portal will be displayed; if impersonating a pharmacy participant, a prescribing test portal will be displayed. Once you’ve made a selection, click Next Step.

Step 3: Sending UID Selection

In order to properly send a test message, you must select the UID (pharmacy or prescriber user ID) from which the message is being sent. This information is controlled by the application’s configuration. You will be notified if there are no valid UIDs from which to send the message.

The drop-down will include the name associated with the UID along with the corresponding NCPDP ID or SPI for tracking purposes. Click Next Step.

Step 4: Receiving UID Selection

Select the appropriate UID from those associated with your portal for the receiving portal to receive the message.

You will be notified if no UIDs are configured for the currently impersonated portal. Note, in order to prevent this from happening, portals listed on the Home page that do not have UIDs will say so next to the portal name. Once the receiving UID has been selected, click Next Step.

Step 5: Patient Profile Selection

It is important to refrain from using PHI (Personal Health Information) for patient profiles.

Step 5 allows the substitution of patient information from a specific profile. The selected profile in the drop-down list limits the patients to only those that belong to the profile in question. If you don’t wish to use a patient profile, select the default value Do Not Use Profile from the Profile drop-down. Click Next Step.

Step 6: Medication Options

Step 6 allows the substitution of medication information from one of the previously created medication entries available to you. You may also choose to use one of the existing (Certification Base) medications that have been generated by the Certification team members. When an account is selected from the drop-down, the medications, if any, will populate for the related account. It is not necessary to use a medication substitution block.

Select the Attach Digital Signature radio button to indicate the message should be digitally signed with a true digital signature. Select the Set Digital Signature Indicator (“SI”) to indicate the message is signed with the “trust me flag” (DRU-110 of SI –Signature Indicator – “signed and archived indicator”). The radio list will default to Not Applicable/None.

The file will fail if Digital Signature is selected but the file does not support it.

Click Next Step and the current message is updated with the selected type of signature.

The digital signature radio boxes are only visible to (directory version or service level).

Step 7: Review of Selections

Step 7 gives you the opportunity to confirm that the values specified for the outgoing test case message are correct. If any values are not what you wanted, click the related Change button in the table to return to the appropriate step so that you can make your changes.

To preview and edit the outbound message body prior to sending the message, expand the Message Body region as shown in the screen print below. If you make edits to the message body, the message will be validated prior to an attempted send. If the newly edited message structure is invalid, an error message will indicate what is wrong with the content. To load up the original message, click Reset To Defaults.

The following image shows a portion of the expanded Message Body section of Step 7 above. Note the Digital Signature portion of the xml has been called out in the image below.

Click Send Message when you are ready to send out the message. The message is processed.

Step 8: Review

When Send Message has been selected, the send results are displayed in this final review page. Transmission Status is expanded by default, displaying a summary of details relating to the attempt.

The expanded Details section (below) contains detailed information regarding the results from the attempted send; this information can be used to determine any issues or status-related data for the message that was sent. Note that this information is also presented to users through the Message Search pages in the Admin Console.

The expanded Message Transmission Response section (below) contains any corresponding status or error message that resulted from the send attempt of the message.

The expanded Sent Message Structure section contains the exact outbound message body that was sent to the portal reflected in the current Impersonation Context.

If you own the test case, you can click Update Test Case to update the existing test case you just sent. This will save any changes made to the existing test case. If you do not own the test case, or if you prefer to clone the test case to a new copy, click Clone Test Case to create a new test case populated with the values from the message that was sent out. Please refer to Edit/Create Test Case on page 1 for more information on cloning a test case.

Response Based Messages

The process for sending Response based messages differs slightly from non-response based messages. Response based message types include: REFRES (EDIFACT for Refill Request) and Error Messages. The difference is that you must select a valid, corresponding Request message which is used to generate the response message based on values from the request.

There are fewer steps in the corresponding Response based message wizard and the overall outcome is the same as that generated from a non-response message.

!! IMPORTANT !!  To send your portal a response message, you must first confirm that at least one corresponding request message has been sent to one of the Test Portals (such as TestXml4Pharmacy or TestXml4Prescriber). The response engine looks for messages that are considered valid in the engine; i.e. the following credentials must be met:
1. Request message sent to testing portal must exist
2. Request message must be valid; i.e. no errors
3. Request message must not have a corresponding response already generated for it

To send a Response Based message, go to the Certification Suite Main page and click the response message test case you wish to execute.

Step 1: Introduction

Step 1 describes the wizard. It is recommended that you confirm you are impersonating the correct portal to which you wish to send the message. Click Next Step.

Step 2: Relative Request Message Selection

Select a valid request message to which you want to generate a response. The messages in the drop-down list are limited by the portal, response message type, availability of the request message (whether or not a response has already been generated for it), and validity of the request message (whether or not it made it through the engine successfully).

You can choose to limit your search results by From Date/Time, To Date/Time and Sender Message ID. If no messages are found, refine the search filters or send at least one valid Request message through the system to one of test partners, as previously described. Once a valid message has been selected, click Next Step.

Step 3: Review of Selections

Confirm that the selected values in the Parameters section specified for the outgoing test case message are correct. If they are not, go back to the corresponding step by pressing the related Change button in the table to make your edits.

To see a preview of the outbound message body prior to sending the message – or to add a digital signature or SI (digital signature indicator) – expand the Outgoing Response Message Body section. Select either the Attach Digital Signature or the Set Digital Signature Indicator radio button. Click the Update Digital Signature button. It would be necessary to select the Update Digital Signature button again if changes were made to any field associated with the digital signature. Please refer to the Prescription Routing Implementation Guide for more detailed information on digital signatures.

When ready to send out the message, click Send Message. The message is processed.

Step 4: Review

When a message has been sent, the results of the attempt are displayed in this final review window. Transmission Status is expanded by default displaying a summary of details relating to the attempt.

The expanded Details section contains detailed information regarding the results from the attempted send. This information can be used to identify any issues or status-related data for the message that was sent. Note that this information is also presented to the users through Message Search in the Admin Console.

The Message Transmission Response section contains any corresponding status or error messages that resulted from the send attempt.

The Sent Message Structure section of the window contains the exact outbound message body that was sent to the portal.

Response message types are built based on the original request and cloning/updating features are not available for these types on this final screen.

Profile Management

A Profile is an organizational unit within the Certification & Testing Suite that contains one or more patients. The hierarchy for profiles is shown below.

Profile Hierarchy

Once a profile has been created, you can create, edit and delete patients within the profile.

Go to Certification Suite>Profile Management. The Profile Manager window opens. This window provides the option to search for profiles by Profile Name and the Last Modified By filters. By default, this page loads the results from a wild-card search, as shown below.

Creating a New / Editing an Existing Profile

To create a new profile, go to Certification Suite>Profile Management and click on the New Profile link at the top of the page.

Enter a profile name in the Name field and additional identifying information as a personal reference in the Profile Description text box. General information about the profile is provided below the text boxes and may not be edited. Once changes are complete, a number of options are available, including View Patients; which redirects you to the Patient Manager for the profile.

To edit a specific profile, go to Certification Suite>Profile Management and click <Profile Name> link for the profile you would like to edit. The Edit/Create Profile window opens, as shown above. Once you have completed your changes to the Name and/or Profile Description, click the Update Profile button to save your changes.

Patient Manager

Once a profile has been created, you may create, edit and delete patients within the profile. At the time of test case execution, you may substitute patient information into the body of the test case, which allows for the use of real world data for the message that is being sent out.

PHI should not be included in patient information.

To view the patients for a profile, from the Profile Manager window shown below, click the Patients link on the far right of the screen or click the View Patients button on the Edit/Create Profile window.

The Patient Manager window opens. Patients are searchable by the Last Name, Social Security Number and Last Modified By fields.

Creating a New/Editing an Existing Patient

To create a new patient within the current profile, click the New Patient link at the top of the screen (as shown above). To edit or delete an existing patient, click on the applicable Patient Name from the search results. The Edit/Create Patient window opens.

The Edit/Create Patient window contains several fields that can be populated with data related to a patient record. Only the Last Name field is required for a patient record, all other information is optional. Any field that is not populated in the patient profile will use the default values from the test case message body.

To access phone numbers for the current patient record, click on the related View Phone Numbers button at the bottom of the screen. The Phone Numbers window opens.

Patient Phone Number Manager

Up to eight phone numbers can be associated with a given patient. No search filters are provided on this page.

Creating a New/Editing an Existing Patient Phone Number

To create a new phone number, click the New Phone Number link at the top of the page. To edit an existing phone number record, click on the actual number of the phone in the search results. The Edit/Create Phone Number window for the selected phone number opens.

In the Phone Number field, enter or change a phone number in the format xxxxxxxxxx (10 digits) where x is an alpha numeric value. Do not include dashes. Next, select a Qualifier for the phone number from the drop-down list. When applicable, the phone numbers for the patient profile will be used to populate any relative phone number block for the patient information at the time of execution.

Medications Management

In certain situations it may be necessary to substitute in specific blocks of medication. The Medications Management section describes how to pre-define medications for substitution in test cases.

To access Medications Management, select the Medications Management icon from the toolbar or the Medications Management link. The Medication Management window opens.

The Medication Management window allows you to search for medications relating to particular accounts (including certification base medications), by Medication Name and Last Modified By.

Editing a Medication

To edit a medication that is associated with your account, click on the medication name. The Edit/Create Medication window opens. When changes are complete, click Update Medication to save your changes.

Cloning a Medication

To clone existing (associated/owned) medications or medications that you do not have edit permissions to (such as Certification Base medication blocks), click the Clone link from the Medication Management window for the medication you wish to clone.

The Edit/Create Medication window opens with values pre-populated from the corresponding medication that was selected for cloning.

Select the corresponding Account from the Account drop-down (defaults to current impersonated account if no other account is selected), type in the Medication Name, and complete any edits. Click the Update Medication button to save the cloned medication.

The Medication XML is based on a standardized xml block that defines a medication. The related XML block contains information that may (depending on message type) be substituted into a message during a send attempt. Please make sure to use the generic tag name, <Medication>. This block of data is validated against a schema to ensure uniformity when the medication is saved. If the XML is invalid, or if a required field is missing, an error message will display.
