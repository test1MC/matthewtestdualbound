Provider Messages

AddProviderLocation

This message can be used to add a new provider or to add an additional unique physical location (Address Line 1) to an existing account. Individual providers may register multiple practice locations.

When adding a new provider:

The provider customer must supply all required information and an SPI is automatically assigned and returned to the customer in the AddProviderLocationResponse message.

When adding a new location to an existing provider:

The provider customer must supply the SPI root in addition to other required information. Once the location is added, Surescripts returns a new SPI that consists of the SPI Root (first 10 digits of the SPI) and the newly generated Location ID (last three digits of the SPI) for the provider that represents the new location. The root will be the same, only the location code will be new.

For example, SPI locations: 123 Main St (SPI-001), 456 Oak St (-002), etc. for the same provider. The same physical AddressLine1 location (e.g. 123 Main St) may be used at more than one of the SPI locations (e.g. 001 and 003) for the provider; however, only one of the SPI locations (001 or 003) with that address (123 Main St) may have the RxRenewal and/or RxChange service levels. An Error message is returned if the data supplied or the structure of the message is invalid.

UpdateProviderLocation

Use the UpdateProviderLocation message to update provider information, such as provider name, demographics, or to activate/deactivate a provider record. This message cannot be used to switch vendors. To switch vendors for a provider, use Admin Console and follow the Provide Update Request (PUR) process.

The information provided in the UpdateProviderLocation message will replace the previous record information. For example, the message will set to null or default any optional element that is not included in the update message. If the submitted update is a name change, the change will be made without proactively notifying other provider vendors of the change(s).

When using UpdateProviderLocation to deactivate a provider location, set the ActiveEndTime to the current date and remove all ServiceLevels from the provider record, or use the DeactivateProviderLocation message.

Best Practice: To avoid unintentional deletion of data, Surescripts recommends implementing provider updates by first using a GetProviderLocation message to obtain all information that should be included in the UpdateProviderLocation message.

All provider record updates must include the Identification element with the SPI of the location being updated.

UpdateProviderLocation - Diagram

GetProviderLocation

The GetProviderLocation message is used to retrieve provider information based on an existing provider's SPI, NPI, DEA Number or State License Number. Surescripts recommends use of this message to obtain real-time information for the following:

  • Prior to provider record owner updating a provider record. Provider details will be returned only if the requesting account (software vendor) owns the provider record.
  • Search for a provider record by an Organization.

To facilitate the PUR process, setting up the Account drop-down to All in Admin Console allows customers to search for providers managed by other accounts.

Error Scenarios:

If the data supplied or the structure of message is invalid, an Error message will be returned.

If there are no provider records found that match the provided identifiers, the system will return a response message indicating that the provider is not found.

Status and Error Response Messages

Some directory messages will return either a Status or Error message, while others that modify or return content will return a related response message. Refer to schema for definitions of these fields.

Status Message

A Status code of "010" indicates that Surescripts received and processed the message successfully.

Status Message-Diagram

Error Message

An Error message will always include a code of "900" indicating that there was an error that needs to be corrected before resending. The message will include DescriptionCode and Description elements that will describe the error. The error could be directory related or a general error message. A full description of Error codes is given in Directory Error Messages.
