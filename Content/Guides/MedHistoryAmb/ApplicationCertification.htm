Application Certification Requirements Summary

Medication History Provider Vendor Requirements

  • The application shall*106M-011 allow the end user to change a patient's consent to obtain medication history between, at a minimum, the yes and no consent values.
  • The application shall*106M-012 display a disclaimer regarding use of medication history retrieved via Surescripts.

Sample Disclaimer: Certain information may not be available or accurate in this report, including items that the patient asked not be disclosed due to patient privacy concerns, over-the-counter medications, low cost prescriptions, prescriptions paid for by the patient or non-participating sources, or errors in insurance claims information. The provider should independently verify medication history with the patient.

  • When a data element from the medication history response is displayed, the application shall*106M-014 have the ability to display all data received for that data element.
  • If the application displays medication history from sources other than Surescripts, the application shall*106M-020 identify which medication history was retrieved via Surescripts.
  • The application shall*106M-018 have the ability to display all medication history MedicationDispensed loops retrieved, unless prohibited by federal, state or local laws.
  • The application shall*106M-019 provide access to all history data available for a specific request. When a medication history response indicates more medication history is available, the application shall*106M-026 either systematically retrieve all medication history available or notify the end user more medication history is available.
  • For PBM/Payer claim data, the customer shall*106M-016 retrieve medication history for eligible coverages only.

For pharmacy fill data, the customer may retrieve medication history regardless of patient eligibility.

  • For PBM/Payer claim data, the customer shall*106M-017 retrieve medication history for all eligible coverages.
  • The claim medication history request for a patient shall*106M-013 be tied to the most recent eligibility response for that patient. This is accomplished by using the ISA control number (ISA-13) from the 271 Eligibility response in the RelatesToMessageID of the RxHistoryRequest.

Medication History PBM/Payer Requirements

MedicationDispensed shall*106M-021 be sent in order of most recent history first according to fill date.

Numeric Representation

The decimal point is represented by a period and shall*106M-001 be used as follows:

  • only when there are significant digits to the right of the decimal
  • when there is a digit before and after the decimal point
  • not with whole numbers

XML Empty Tags

Customer shall*106M-005 not send empty elements within XML messaging with exception of the RxHistoryResponse/Approved element.

All non-essential characters shall*106M-002 be suppressed from the message where permissible. Non-essential characters include leading spaces, trailing spaces, leading zeroes, and trailing zeroes where permissible (see Numeric Representation).

Handling Optional Fields

The customer shall*106M-007 be able to process any valid incoming message request or response, including syntactically valid maximum and minimum population messages. Optional data elements (and values therein) shall*106M-008 not cause message failure.

Error Message

For PBM/Payers, an asynchronous error message (also called Free Standing Error message) is sent in response to an invalid RxHistoryResponse message. The asynchronous error message is a separate message type and shall*106M-022 be responded to with a status of 010. An Error message shall*106M-023 never be responded to with an Error message; this could create an endless loop.

Message Elements

  • MessageID: Each message sent from a customer shall*106M-009 have a unique Message ID. MessageIDs should not be reused for 18 months.
  • RelatesToMessageID: The MessageID shall*106M-010 be echoed back in the response to the message that was received.
  • Number: Phone Number shall*106M-025 be in the format 5552223333X4444 - where 555 is the area code, 2223333 is the main number and X4444 is the extension. Phone number shall*106M-028 not be populated with all zeros or other default values. This field is 80 characters long to handle email addresses. (Please note the extension X4444 is just an example. Less/more than four digits are allowed).
  • DrugDescription: The item or drug description field shall*106M-031 include the full drug name, strength and form (in that exact sequence).
  • DrugCoded/ProductCode: When sending, the NDC must be an 11 digit NDC in the 5-4-2 format. Dashes shall*106M-032 not be used and leading zeros shall*106M-033 not be suppressed.

Example: 12345123412. When receiving a message, the NDC may be blank, NDC9, or NDC10 because it was translated from an older version.

  • StrengthCode: If either or both the Drug Coded Strength and the Strength Code are sent they shall* 106M-034 match the Drug Strength in the Drug Name Description field. Drug strength units should not be sent in free text Drug Strength field.
  • DEASchedule: If the medication is categorized as a controlled substance by the Drug Enforcement Agency (DEA), this field shall*106M-027 be sent.

XML

Most symbols should be 'safe' within most alphanumeric fields; however, XML validation has issue with the following characters and they shall*106M-035 be escaped. The most common escape encodings are:

ASCII Characters

XML Escape

quote (")

&quot;

apostrophe (')

&apos;

ampersand (&)

&amp;

less than (<)

&lt;

greater than (>)

&gt;

String lengths are calculated based on the un-escaped value. For example, “>98 degrees” should be encoded as “&gt;98 degrees” but would only be counted as a string length of 11, not 14.
