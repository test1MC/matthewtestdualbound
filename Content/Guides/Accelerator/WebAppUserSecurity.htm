User Security

Users do not log into the Accelerator separately from their provider vendor system. The provider vendor is responsible for authenticating the user and determining what data and activities users will have access to. The provider vendor system specifies the User ID and permissions in the Auth Token Request. The Accelerator limits information and activities available to those specified for the user in the token. In response, the Accelerator presents only the information and activities authorized for that user.

At a high level, the provider vendor and the Accelerator share responsibilities for maintaining and enforcing user authentication and access controls for product functions. Responsibilities are described below.

The provider vendor:

  • maintains its users, their roles, their relationships (e.g., delegation of authority from providers to staff), and their authentication credentials, etc.
  • includes pertinent user information when initiating the Accelerator's user interface and task list components, including:
    • the User ID (as defined by the provider vendor system)

      • The provider vendor must ensure the User ID is unique across their customer base.

    • an assertion that the user has been authenticated by the provider vendor system
    • the specific roles the user possesses
    • the providers on whose behalf the user can act

The Accelerator:

  • logs the User ID (as defined by the provider vendor system) and activity
  • enforces the user’s role(s) assigned by the provider vendor system
  • enforces visibility of data according to the delegation information supplied by the provider vendor system

Authentication and Authorization

Users are authenticated by the provider vendor system. The Accelerator tracks the User ID of the individual signed into the provider vendor system.

The provider vendor system authorizes a user to perform a task by assigning them to a role. The types of roles are: The type of role is:

  • PA Preparer: The PA Preparer has create, read, and edit permissions.
  • PA Reviewer: The PA Reviewer has read-only permissions.
  • PA Submitter: The PA Submitter has create, read, edit, and submit permissions.
  • Viewer: The Viewer has read-only permissions.
  • Submitter: The Submitter has create, read, edit, and submit permissions.
  • PN.Viewer: The PN.Viewer has limited permissions.
  • SPO Reviewer: The SPO Reviewer has read-only permission.
  • SPO Submitter: The SPO Submitter has create, read, edit, and submit permissions.

The user may have different roles depending on their relationship to the prescriber (e.g., a user may have the PA SubmitterSubmitter role when working on behalf of Dr. Smith and the PA Reviewer Viewer role when working on behalf of Dr. Jones).

The user may have different roles depending on their relationship to the provider (e.g., a user may have the SPO Submitter role when working on behalf of Dr. Smith and the SPO Reviewer role when working on behalf of Dr. Jones).

The users and their roles are not stored by Surescripts. The provider vendor system must securely communicate the authenticated User ID and roles to Surescripts.
