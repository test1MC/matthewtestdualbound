Roles

A list of roles is required in order to generate an auth token. The provider vendor system assigns a role to the user, which authorizes them to perform certain tasks within the Accelerator. The provider vendor system requests an auth token, which specifies role requests for each launch of the Accelerator UI. The Accelerator encodes the requested roles into the auth token. See Authentication and Authorization for role(s) authorized to perform specific tasks.

Roles can be specified per the following:

  • A common role can be specified across all providers assigned to that user.
  • Roles can be specified on a role-per-provider basis or on a list of providers basis.

    • The list of providers the user can access is managed by the provider vendor system.

  • Roles can be specified on a per-patient basis or on a list of patients basis.

    • The list of patients is managed by the provider vendor system.

If an attempt to create an auth token violates one of the following rules, the Accelerator will return a validation error. The rules include:

  • Roles must be assigned at the common level, or the provider level., or the patient level.
  • Levels cannot be combined and at least one level must be specified.
  • When specifying providers, a provider may not be specified more than once.
  • When specifying patients, a patient may not be specified more than once.

Roles Definitions

Roles

Notes

Common Roles

An array of Roles that apply to all submitted providers or patients. When Common Roles are included in the request, Roles assigned for Providers and Patient will be ignored and should not be sent.

Providers

An array of Provider objects, each with an array of roles for that provider, an id, and a qualifier.

Patient

An object that describes a patient with an alternateId, an alternateOrganization, alternateQualifierid, an organization, and an array of roles that apply.

Patient

An object that describes a patient with an id, an organization, and an array of roles that apply.

Role per Patient Assignment

The following example shows how a specific role can be assigned for a specific patient.

In addition to the table at the beginning of this section, the table below details additional JSON fields specific to this example.

JSON Fields

Notes
alternateId Corresponds to the patient’s PBM Unique Id returned in the 271 eligibility response.
alternateOrganization Corresponds to the payer Id returned in the 271 eligibility response.
alternateQualifier

Qualifier for the patient identifier.

alternateQualifier: PBMUID

id

Corresponds to the MedicalRecordIdentificationNumberEHR field (...Patient/Identification/MedicalRecordIdentificationNumberEHR) specified in the PAInitiationRequest.

Matches the MedicalRecordIdentificationNumberEHR field (...Patient/Identification/MedicalRecordIdentificationNumberEHR) specified in the PAInitiationRequestSPOInitiationRequest.

This field is optional in the schema, but has to be included in the original PAInitiationRequestSPOInitiationRequest in order for the resulting process to be visible when launching the Accelerator for a specific patient.

organization

The value sent in the MutuallyDefined field of the PAInitiationRequest. Used to identify a patient with the same ID across different locations/facilities.

The value sent in the MutuallyDefined field (...Patient/Identification/MutuallyDefined) of the PAInitiationRequestSPOInitiationRequest. Used to identify a patient with the same ID across different locations/facilities.

Note: If this optional field is not included in the PAInitiationRequests, it cannot be included in auth token requests for the specified patient. Without “organization”, all PAs for the specified patient will display regardless of location/facility.

Note: If this optional field is not included in the PAInitiationRequestSPOInitiationRequest, it should not be included in Auth Token Request for the specified patient. The worklist items will show when patient id equals MedicalRecordIdentificationNumberEHR and organization equals MutuallyDefined. For example, if organization is "null" and MutuallyDefined is "test clinic", the task for that PAInitiationRequestSPOInitiationRequest will not be displayed.
